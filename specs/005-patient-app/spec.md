# Feature Specification: Carelog 환자 앱 — 상담 내역 수신·가입·서비스 이용 흐름

**Feature Branch**: `005-patient-app`
**Created**: 2026-05-17
**Status**: Draft

## User Scenarios & Testing *(mandatory)*

### User Story 1 - SMS 수신 후 상담 내역 확인 및 가입 유도 (Priority: P1)

치과에서 상담이 끝나면 직원이 "상담 공유" 버튼을 누르고, 환자 핸드폰에 SMS가 도착한다. 링크를 탭하면 주민번호 앞자리 + 전화번호로 본인 인증 후 오늘의 상담 내역이 표시된다. 상담 기록 하단에는 "Carelog에 가입하면 모든 진료 기록을 앱에서 언제든 확인할 수 있습니다"라는 CTA가 보이고, 카카오 또는 Google 소셜 로그인으로 1분 이내에 가입을 완료할 수 있다.

**Why this priority**: 환자의 첫 접점이자 전환 기회. 이 흐름이 없으면 환자 앱 자체가 성립하지 않는다.

**Independent Test**: 직원이 상담 공유 → SMS 수신 → 링크 클릭 → 인증 → 상담 내역 확인 → 소셜 로그인으로 가입 완료 → "가입 완료" 화면 표시.

**Acceptance Scenarios**:

1. **Given** 직원이 환자 상세 페이지에서 상담 공유 버튼을 누르면, **When** SMS가 발송되면, **Then** 환자는 "오늘 진료 기록 보기" 링크가 포함된 문자를 받는다.
2. **Given** 환자가 SMS 링크를 탭하면, **When** 주민번호 앞자리 + 전화번호를 입력하면, **Then** 오늘 상담 내용, 처치 요약, 처방/권고 사항이 표시된다.
3. **Given** 환자가 상담 내역을 확인한 후, **When** 가입 CTA를 탭하면, **Then** Google 소셜 로그인 화면으로 이동한다.
4. **Given** 소셜 로그인을 완료하면, **When** 환자 계정이 생성되면, **Then** 해당 치과의 모든 과거 진료 기록이 즉시 계정에 연결된다.
5. **Given** 가입이 이미 완료된 환자가 새 SMS 링크를 받으면, **When** 링크를 탭하면, **Then** 인증 없이 소셜 로그인만으로 바로 접근할 수 있다.

### User Story 2 - 환자 앱에서 전체 진료 이력 조회 (Priority: P2)

가입한 환자는 Carelog 앱(PWA)을 홈 화면에 추가하고, 앱을 열면 소셜 로그인으로 바로 접근한다. 앱 안에서 자신이 다닌 모든 치과의 진료 기록을 날짜순으로 확인할 수 있다. 각 기록에는 기관명, 상담 내용, 처방/권고 사항이 표시된다.

**Why this priority**: 앱을 재방문하게 만드는 핵심 가치. 진료 이력이 쌓일수록 Carelog의 가치가 높아진다.

**Independent Test**: 가입 완료 환자가 앱 로그인 → 전체 진료 이력 목록 표시 → 각 기록 상세 확인 가능.

**Acceptance Scenarios**:

1. **Given** 가입한 환자가 앱을 열면, **When** 소셜 로그인을 완료하면, **Then** 연결된 모든 기관의 진료 기록이 날짜 역순으로 표시된다.
2. **Given** 진료 기록 목록에서 항목을 탭하면, **When** 상세 화면이 열리면, **Then** 기관명, 날짜, 상담 내용, 처방/권고 사항이 표시된다.
3. **Given** 환자가 여러 치과를 다닌 경우, **When** 기록 목록을 보면, **Then** 기관별 구분 표시와 함께 통합 목록이 제공된다.

---

### User Story 3 - 새 상담 기록 생성 시 환자에게 푸시 알림 전송 (Priority: P3)

치과 직원이 환자의 상담 기록을 저장하면, 해당 환자가 Carelog에 가입되어 있고 알림을 허용한 경우, 환자의 기기로 푸시 알림이 발송된다. 알림을 탭하면 새 상담 기록 화면으로 바로 이동한다.

**Why this priority**: 환자가 앱을 능동적으로 열지 않아도 새 기록을 인식하게 하는 리텐션 장치.

**Independent Test**: 직원이 상담 저장 → 5초 내 환자 기기에 푸시 알림 도착 → 탭 시 해당 기록 화면 이동.

**Acceptance Scenarios**:

1. **Given** 가입 환자가 알림을 허용한 상태에서 직원이 상담 기록을 저장하면, **When** 저장이 완료되면, **Then** 5초 이내에 환자 기기로 "새 진료 기록이 등록되었습니다" 푸시 알림이 도착한다.
2. **Given** 알림을 탭하면, **When** 앱이 열리면, **Then** 해당 상담 기록 상세 화면으로 이동한다.
3. **Given** 환자가 알림을 허용하지 않은 경우, **When** 상담 기록이 저장되면, **Then** 알림 없이 앱에서 직접 확인만 가능하다.

---

### User Story 4 - 이중 역할: 직원이자 환자로서의 경험 (Priority: P4)

같은 사람이 Carelog에서 직원(치과 스태프)으로도, 환자로도 활동할 수 있다. 직원 계정으로 로그인하면 상담 기록을 입력하고 환자를 관리하는 치과 업무 화면이 보이고, 환자 계정으로 전환하면 본인의 진료 기록을 확인하는 환자 앱 화면이 보인다. 두 계정은 동일 이메일/소셜 계정으로 운영 가능하다.

**Why this priority**: 최고 관리자(송정현)가 서비스를 직접 체험하고 검증하기 위해 필요. 또한 치과 직원 자신이 그 치과의 환자이기도 한 현실적인 케이스를 지원한다.

**Independent Test**: 직원 계정 로그인 → 상담 기록 입력 → "내 진료 기록 보기" 전환 → 환자 앱 화면으로 이동 → 본인의 진료 기록 확인.

**Acceptance Scenarios**:

1. **Given** 직원이 로그인한 상태에서, **When** 헤더의 "내 진료 기록" 메뉴를 탭하면, **Then** 환자 앱 화면으로 전환되어 본인의 진료 기록이 표시된다.
2. **Given** 환자 앱 화면에서, **When** "직원 화면으로 돌아가기"를 탭하면, **Then** 이전에 보던 직원 대시보드로 돌아간다.
3. **Given** 직원이 환자 계정을 아직 연결하지 않은 경우, **When** "내 진료 기록"을 탭하면, **Then** 본인 인증(주민번호+전화번호) 후 환자 계정을 연결하도록 안내한다.

---

### Edge Cases

- 동일 주민번호+전화번호로 여러 치과에 환자로 등록된 경우 → 하나의 환자 계정에 모든 기관 기록 통합 연결
- 가입 시 사용한 소셜 계정이 이미 직원 계정으로 사용 중인 이메일인 경우 → 두 역할이 동일 소셜 계정에 연결 (구분은 역할로만)
- 환자가 소셜 로그인 후 계정 연결 도중 이탈한 경우 → 재접속 시 연결 완료 안내
- SMS 링크 만료(72시간 초과) 후 접근 시 → 만료 안내 + 직접 앱 로그인 유도
- 환자가 여러 소셜 계정(카카오, Google)을 가진 경우 → 하나의 환자 계정에 여러 소셜 계정 연결 가능
- 환자가 상담 공유 SMS를 받지 못한 경우 → 앱 로그인 후 직접 기록 조회 가능

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 환자가 Google 소셜 로그인으로 Carelog 환자 계정을 생성할 수 있어야 한다. (카카오는 추후 추가)
- **FR-002**: 소셜 로그인 시 기존 주민번호+전화번호 인증 기록과 자동 연결되어야 한다.
- **FR-003**: 환자 앱은 직원 앱과 별도의 화면 구조를 가지며, 환자에게 최적화된 UI를 제공해야 한다.
- **FR-004**: 환자 앱에서 연결된 모든 기관의 진료 기록을 날짜 역순으로 조회할 수 있어야 한다.
- **FR-005**: 각 진료 기록에는 기관명, 날짜, 상담 내용 요약, 처방/권고 사항이 표시되어야 한다.
- **FR-006**: 가입한 환자에게 새 상담 기록 저장 시 푸시 알림을 발송할 수 있어야 한다.
- **FR-007**: 환자는 알림 수신 동의/해제를 앱 내에서 관리할 수 있어야 한다.
- **FR-008**: 환자 앱은 PWA로 홈 화면에 추가하여 앱처럼 사용할 수 있어야 한다.
- **FR-009**: 직원 계정을 가진 사용자가 환자 계정으로도 전환하여 본인의 진료 기록을 확인할 수 있어야 한다.
- **FR-010**: 동일 소셜 계정(이메일)이 직원과 환자 두 역할을 동시에 가질 수 있어야 한다.
- **FR-011**: 가입 이후에는 소셜 로그인만으로 환자 앱에 접근할 수 있어야 한다 (반복 주민번호 인증 불필요).
- **FR-012**: SMS 링크를 통한 최초 접근 시 인증 완료 후 가입 CTA가 표시되어야 한다.
- **FR-013**: 상담 내역 화면은 환자가 이해하기 쉬운 언어와 레이아웃으로 구성되어야 한다.

### Key Entities

- **환자 계정(Patient Account)**: 소셜 로그인과 연결된 환자 영구 계정. 주민번호 해시, 전화번호, 연결된 소셜 제공자(카카오/Google), 푸시 구독 정보
- **환자-기관 연결(Patient-Institution Link)**: 특정 환자 계정이 어느 기관의 어느 patient 레코드와 연결되는지 매핑. 한 환자가 여러 기관에 연결 가능
- **이중 역할 연결(Dual Role Link)**: 직원 계정과 환자 계정이 동일 소셜 계정으로 연결된 경우를 추적

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: SMS 수신부터 가입 완료까지 3분 이내 수행 가능
- **SC-002**: 가입 후 모든 과거 진료 기록이 30초 이내에 계정에 연결되어 표시
- **SC-003**: 재방문 시 소셜 로그인으로 10초 이내 앱 접근 완료
- **SC-004**: 상담 저장 후 5초 이내 환자 기기에 푸시 알림 도달
- **SC-005**: 직원 화면 ↔ 환자 앱 전환이 2초 이내 완료
- **SC-006**: 환자 앱에서 진료 기록 목록 로드가 3초 이내 완료
- **SC-007**: SMS 링크를 통한 최초 접근 → 가입 전환율 목표 30% 이상 (추후 측정)

## Assumptions

- v1에서는 Google 소셜 로그인만 지원한다. 카카오는 추후 별도 스펙으로 추가한다.
- Google 로그인은 이미 구현된 직원용 Google OAuth와 동일한 프로바이더를 활용한다.
- 환자 앱과 직원 앱은 동일 도메인(carelog.app)에서 URL 경로로 구분한다 (`/portal/` vs `/`).
- 환자의 주민번호 해시(rrn_hash)가 계정 연결의 기준이 되며, 이미 시스템에 존재한다.
- 환자 앱 전용 화면은 기존 `/portal/` 경로를 확장하는 방향으로 구현한다.
- 푸시 알림은 직원 알림과 별도로 환자 전용 push_subscriptions 채널을 사용한다.
- 이중 역할 사용자(직원이자 환자)는 동일 브라우저에서 두 역할 모두 사용 가능하다.
- 상담 내용의 민감 정보(주민번호 등)는 환자 앱에서도 동일하게 마스킹 처리된다.
