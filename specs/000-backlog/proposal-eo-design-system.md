# 제안서 — 디자인·UX 공통화 및 제품군 연계 체계

> **상태**: 2026-05-31 실비(EO 기획)에게 전달 완료 · EO 총괄/기획 검토 대기
> **작성**: 다온 (Carelog 기획·개발 겸임)
> **대상**: EO 총괄 매니저 / 기획 리드
> **목적**: EO를 허브로 Carelog·Iris가 같은 디자인·UX 언어를 상속하는 경량 연계 체계 합의

---

**받는 분**: EO 총괄 매니저 / 기획 리드
**보내는 이**: 다온 (Carelog 기획·개발 겸임)
**일자**: 2026-05-31
**한 줄 요약**: EO를 허브로, Carelog·Iris가 같은 디자인·UX 언어를 상속해 **"하나의 제품군"**으로 보이게 하되, 각 제품의 git·배포·DB 격리는 그대로 유지하는 **경량 연계 체계**를 제안합니다.

---

## 1. 배경

- 현재 EO / Carelog / Iris는 각자 별도 git·Vercel·Supabase에서 독립 운영 중입니다. (데이터·배포 격리 측면에선 바람직)
- 다만 사용자 입장에선 **세 제품이 따로 노는 인상**을 줄 수 있어, 브랜드 자산이 분산됩니다.
- Carelog에 온보딩/가이드 기능을 넣으려는 시점에서, **이걸 제품마다 따로 만들지, 공통 자산으로 만들지**를 지금 정하면 향후 중복을 크게 줄일 수 있습니다.

## 2. 제안 핵심

> **EO를 상위 플랫폼(허브)으로 두고, Carelog·Iris는 그 산하 제품으로 같은 디자인 시스템을 상속한다.**
> 단, 결합은 **디자인 레이어에만 한정** — 데이터·인증·배포는 계속 분리해 각 제품의 독립 상품화 가능성을 보존한다.

Carelog는 **"EO의 서브 프로젝트"라는 정체성으로 진행해도 좋고**, 동시에 단독 제품으로도 손색없게 끌고 갈 수 있는 구조입니다. 두 길이 배타적이지 않게 설계했습니다.

## 3. 무엇을 공유하나 — 3개 레이어

| 레이어 | 공유 대상 | 결합도 | EO 역할 |
|--------|-----------|--------|---------|
| **1. 디자인 시스템** | 토큰(색·간격·타이포·radius), 코어 컴포넌트, 브랜드 | 중 | **소스 오브 트루스(본가)** |
| **2. 아이덴티티·내비** | 앱 스위처, 공통 헤더/푸터, 도메인, (후순위)SSO | 중~높음 | 허브 진입점 |
| **3. 거버넌스** | 디자인 헌법(원칙·보이스&톤), 의사결정 방식 | 낮음 | 원칙 제정 주체 |

## 4. 결합 방식 (기술 부담 최소화)

- **토큰 = 강결합(공유 의존성):** 색·간격은 100% 동일하게. EO 기준 토큰을 패키지로 배포 → Carelog·Iris가 install.
- **컴포넌트 = 느슨하게(복사·동기화):** 제품별 진화 속도가 달라도 안 깨지게, shadcn식으로 복사해 쓰고 스크립트로 동기화.
- **런타임/DB 무관:** 디자인 시스템은 빌드타임 의존성일 뿐, 각 Supabase·인증과 무관 → **격리 그대로 유지.**

## 5. EO에 드리는 요청 (The Ask)

1. **허브 오너십 합의** — 공통 디자인 시스템 레포(가칭 `eo-design` 또는 `suwant-design`)의 **본가를 EO 측으로 둘지** 결정.
2. **EO 디자인 토큰·원칙 제공** — 현재 EO의 색/타이포/간격 체계와 디자인 원칙을 공통 토큰의 기준선으로 삼게 공유.
3. **앱 스위처 등재** — EO 헤더에 Carelog·Iris로 가는 전환 메뉴를 둘 의향 확인.
4. **(후순위) 도메인·SSO 논의** — 서브도메인 통일(`*.eo.io` 등), 통합 로그인은 필요해질 때 별도 안건으로.

## 6. EO가 얻는 것 (왜 EO에 이득인가)

- **포트폴리오 브랜드 자산화** — 3제품이 한 우산 아래 보이면 EO 브랜드 파워가 곱해집니다.
- **신규 프로젝트 부팅 속도** — Iris를 포함, 앞으로 만들 모든 제품이 디자인 시스템을 상속해 **0일차부터 EO 룩앤필**로 출발.
- **중복 제거** — 가이드/온보딩 같은 공통 UX를 한 번 만들어 3제품이 공유.
- **EO가 표준의 중심** — 디자인 결정권이 EO에 모여, 일관성의 구심점이 됩니다.

## 7. 단계적 로드맵

| Phase | 내용 | 주체 |
|-------|------|------|
| **0** | 디자인 헌법 초안 + 각 제품 토큰 추출(현황 정리) | Carelog 선행 → EO 검토 |
| **1** | 공통 레포: 토큰 + Tailwind preset, 3앱 연결 | EO(본가) + 각 제품 |
| **2** | 코어 컴포넌트 + 앱 스위처 + 가이드 컴포넌트 | 공통 |
| **3** | 도메인 통일(서브도메인) | EO 주도 |
| **4** | (필요 시) SSO 중앙화 | EO 주도 |

**가장 효과 큰 구간은 Phase 1 + 앱 스위처** — 적은 노력으로 "한 서비스" 체감의 대부분을 확보합니다.

## 8. 안전장치 (리스크 관리)

- 디자인만 공유, **데이터·인증·배포는 영구 분리** → 한 제품 장애가 다른 제품에 전파되지 않음.
- 컴포넌트는 느슨 결합 → **버전 락스텝 부담 없음.**
- Carelog는 서브 프로젝트로도, 단독 제품으로도 갈 수 있는 양립 구조 유지.

## 9. 정해주시면 좋을 것

1. 공통 디자인 시스템 **본가를 EO로 둘지** (권장) / 중립 레포로 둘지
2. Carelog의 포지셔닝 — **EO 서브 프로젝트로 명시** / 느슨한 제휴 제품
3. Phase 0(헌법·토큰 추출)을 **Carelog가 선발대로 먼저 진행**하는 것에 대한 동의

먼저 Phase 0는 Carelog 쪽에서 부담 없이 착수해 초안을 만들어 공유드리겠습니다. 방향성만 확인해주시면 본격화하겠습니다.
