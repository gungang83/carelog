> 출처: 오늘의 치과(서범기) 2차 미팅 O-1 (`pilot-ohneul-260624.md`). 대표 "오늘 방향성 결정."

## 1. 현재 케어로그 STT 파이프라인 (코드 근거)

| 단계 | 구현 | 위치 |
|---|---|---|
| 녹음 | `MediaRecorder`(`audio/webm;codecs=opus`), 250ms 청크 누적 → **단일 blob** | `chair-provider.tsx:279~294` |
| 전사 | OpenAI **`whisper-1`**, `language:"ko"` 고정, `response_format:"text"` | `app/actions/transcribe.ts:31~36` |

**핵심**: 현재는 ① 한국어 고정 ② **사후 일괄(batch)** — 실시간 스트리밍이 아님 ③ 번역 없음.
→ O-1은 이 파이프라인의 **언어 확장 + (선택)실시간화** 두 축.

## 2. 가능 수준을 두 레벨로 분해

서범기 발언이 정확히 두 레벨을 가른다: *"동시통역 말고 녹음하는 거라도 되면"* + *"원문 위, 통역 아래, 버튼 토글."*

### Level 1 — 사후 다국어 전사·번역 (post-hoc bilingual) 🟢 낮은 위험
- 녹음 종료 후: 전사(언어 자동 감지) → **원문 보존 + LLM 번역 패스** → 원문/번역 쌍 렌더.
- **현재 파이프라인 거의 그대로 재사용.** 바뀌는 것:
  - `whisper-1 language:"ko"` 고정 제거 → **자동 감지** 또는 기관 설정 언어. (Whisper는 다국어 전사 지원. `translations` 엔드포인트는 영어로만 번역되므로 ❌ — 대신 전사는 원문 유지하고 **번역은 Claude/GPT 패스**로: 임의 언어쌍·의료용어 프롬프트 제어 가능.)
  - 출력 스키마: `{ original, originalLang, translated, targetLang }` → 보드에 원문/번역 토글.
- **난이도**: 소. **추가 비용**: 번역 LLM 호출 1회(저렴). **리스크**: 거의 없음(기존 검증된 경로).
- **가치**: 서범기가 "이거라도 되면" 한 바로 그 효용. 외국인 상담 기록·전달 즉시 가능.

### Level 2 — 실시간 동시통역 (live streaming interpretation) 🟡 중간 위험·높은 가치
- 대화 중 마이크 스트림 → **스트리밍 STT(부분 결과 ~수백 ms)** + 점진 번역 → 화면 실시간 갱신(원문/번역 2단).
- 현재 `MediaRecorder` 단일 blob 구조를 **WebSocket 스트리밍**으로 확장 필요.
- **2026 기준 전용 모델이 존재** → 기술적으로 가능(아래 §3). 다만 현실 제약(§4)이 관건.

## 3. 2026-06 외부 기술 수준 (조사 결과)

실시간 음성 번역은 2026년 **전용 모델·API가 성숙**. 핵심 후보:

| 제공자 | 강점 | 케어로그 적합성 |
|---|---|---|
| **Soniox** | 실시간 STT+번역, **문장 중간 코드스위칭**(한↔영, 일↔영 혼용) 자동 감지, **sub-200ms**, 화자 분리(diarization), 도메인 용어, 60+ 언어 | ★ 치과 = 한·영 혼용 상담이라 **코드스위칭이 결정적**. 가장 적합 |
| **OpenAI `gpt-realtime-translate`** | 라이브 음성→음성+텍스트, 소스 자동 감지, 70+ 입력→13 출력(한·영·일 포함), **~$0.034/분 정액**, WebSocket(24kHz PCM16). 동사까지 듣고 번역(자연스러움 우선) | 우리가 이미 OpenAI 사용 → 통합 쉬움. 단 코드스위칭은 Soniox보다 약할 수 있음 |
| Azure Speech Translation / Gemini Live / Deepgram / AssemblyAI | 스트리밍 STT·번역 KO/EN/JA 지원 | 대안. 벤치마크 비교 필요 |

**결론**: Level 2도 **2026년엔 자체 모델 개발 불필요** — 대표 철학("기술 주도 말고 적용") 그대로. 적용·통합·UX가 우리 일.

## 4. 핵심 기술 리스크 & 현실 제약 (면밀 검토)

| # | 리스크 | 평가 / 완화 |
|---|---|---|
| R1 | **코드스위칭**(원장이 영어 상담 중 한국어 섞음, 환자 영어) | 치과 현실의 핵심. Soniox가 명시적 지원 → provider 선택의 1순위 기준 |
| R2 | **의료/치과 용어**(인레이·크라운·크랙·차트번호) | 도메인 용어 인식·번역 정확도. 커스텀 어휘/프롬프트 보정. **사람 검수 전제**(케어로그는 보조, 최종은 사람) |
| R3 | **소음·저음량 화자** (예미안 C-09 = 진 원장 작은 목소리) | 하드웨어(O-6 마이크)·VAD 의존. 통역 정확도는 입력 품질에 종속 — 과약속 금지 |
| R4 | **화자 분리**(원장/환자/실장) | diarization 필요(Soniox 지원). 없으면 원문이 누구 말인지 모호 |
| R5 | **모바일 WebSocket 안정성** | 체어=PC 권장(O-6)이라 완화. 모바일 백그라운드 정지(기존 iOS 이슈)와 동일 제약 |
| R6 | **API 키 노출**(브라우저→실시간 API 직결 시) | **서버 릴레이 or ephemeral token** 필수. OpenAI Realtime은 ephemeral key 지원 |
| R7 | **비용**(실시간은 분당 과금) | ~$0.034/분 → 30분 외국인 상담 ≈ $1. **토큰 과금 모델(전사 1분=토큰1)에 자연 편입** → 다국어=Pro/Enterprise 프리미엄 토큰 기능으로 수익화 |
| R8 | **헌법 I(PII·음성 격리)** | 실시간 통역은 **해당 상담 화면 내 표시**(기관 broadcast 아님) → C-05와 달리 PII 확산 없음. 외부 API 전송은 기존 Whisper와 동일 수준. 음성 보관은 spec 009 정책 준수 |

→ **치명적 막힘 없음.** R1·R2가 품질의 핵심이고, "보조 도구·사람 최종검수" 포지셔닝으로 과약속 위험 관리.

## 5. 권장 단계적 로드맵

### Phase 1 — 사후 다국어 전사·번역 (Level 1) ⏭ 즉시 착수 가능
- `transcribe.ts` 언어 자동감지 + Claude 번역 패스 → `{원문, 번역}` 반환.
- 보드/카드에 **원문/번역 토글**(서범기 "원문 위·통역 아래·버튼"). 기관 기본 대상언어 설정.
- DB: consultation에 번역 필드(또는 본문에 병기). 마이그레이션 소.
- **산출**: 외국인 상담을 즉시 이중언어 기록·전달. 위험 낮고 가치 즉시.

### Phase 2 — 실시간 통역 파일럿 (Level 2, provider 검증)
- Soniox vs OpenAI realtime-translate **PoC 비교**(코드스위칭·치과용어·지연·비용).
- 스트리밍 경로: 마이크 → (서버 릴레이/ephemeral) → 스트리밍 STT+번역 → 보드 라이브 2단 표시.
- 오늘의 치과 외국인 신환 1~2건 실측 → 정확도·체감 피드백.

### Phase 3 — 프로덕션·수익화
- 토큰 과금 편입(R7), Pro/Enterprise 게이트. 마케팅(외국인 상담 영상) 자산화.
- 화자 분리·용어 사전·다국어 환자 전달 홈까지.
