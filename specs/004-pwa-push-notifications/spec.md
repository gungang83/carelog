# Feature Specification: Carelog PWA — 홈 화면 추가·푸시 알림·항시 로그인·헤더 고정·푸터

**Feature Branch**: `004-pwa-push-notifications`
**Created**: 2026-05-17
**Status**: Draft

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 홈 화면에 앱 추가 및 앱처럼 실행 (Priority: P1)

치과 직원이 스마트폰이나 태블릿의 브라우저에서 Carelog를 열면 "홈 화면에 추가" 안내가 표시된다. 설치 후 홈 화면 아이콘을 탭하면 네이티브 앱처럼 전체 화면으로 실행되며, 주소 표시줄·브라우저 UI 없이 사용할 수 있다.

**Why this priority**: 앱처럼 접근할 수 있어야 푸시 알림 등 후속 기능의 기반이 되며, 직원 업무 동선에서 가장 빠른 접근을 제공한다.

**Independent Test**: 모바일 브라우저에서 Carelog 접속 후 홈 화면 추가 → 아이콘 탭 → 전체 화면 실행 및 Carelog 아이콘 표시 확인.

**Acceptance Scenarios**:

1. **Given** 모바일 브라우저에서 Carelog를 열면, **When** "홈 화면에 추가" 배너 또는 메뉴가 표시될 때, **Then** 사용자가 설치 절차를 완료할 수 있다.
2. **Given** 홈 화면에 추가된 아이콘을 탭하면, **When** 앱이 실행될 때, **Then** 브라우저 주소 표시줄 없이 전체 화면으로 Carelog가 열린다.
3. **Given** 앱이 전체 화면으로 실행되면, **When** 화면을 스크롤하면, **Then** Carelog 로고·아이콘이 스플래시 화면으로 표시된 후 메인 화면으로 이동한다.

---

### User Story 2 - 항시 로그인 유지 (Priority: P2)

앱을 닫았다가 다시 열어도 로그인 상태가 유지된다. 장시간 미사용 후에도 세션이 자동으로 갱신되어 재로그인 없이 바로 사용할 수 있다. 세션 갱신이 불가능한 경우에만 로그인 화면으로 이동한다.

**Why this priority**: 업무 중 앱을 열 때마다 로그인해야 한다면 직원 경험이 크게 저하된다.

**Independent Test**: 앱 실행 후 24시간 이상 방치 → 재실행 시 로그인 화면 없이 바로 메인 화면 표시 확인.

**Acceptance Scenarios**:

1. **Given** 로그인된 상태에서 앱을 닫고 재실행하면, **When** 앱이 로드될 때, **Then** 로그인 화면 없이 메인 화면이 바로 표시된다.
2. **Given** 세션이 만료될 시점에 앱을 사용 중이면, **When** 세션 갱신이 백그라운드에서 진행되면, **Then** 사용자는 중단 없이 계속 사용할 수 있다.
3. **Given** 세션 갱신이 실패(네트워크 오류, 토큰 무효화 등)하면, **When** 앱이 재실행되면, **Then** 친절한 안내 메시지와 함께 로그인 화면으로 이동한다.

---

### User Story 3 - 푸시 알림 수신 (Priority: P3)

치과 직원이 앱을 사용하지 않는 중에도, 새로운 상담 기록이 저장되거나 활동 이벤트가 발생하면 기기 알림(푸시)을 받는다. 알림을 탭하면 해당 기록 화면으로 바로 이동한다.

**Why this priority**: 실시간 알림은 팀 내 정보 공유와 즉각 대응에 핵심이나, PWA 기반 설치가 선행되어야 한다.

**Independent Test**: 상담 기록 저장 → 앱이 백그라운드 상태인 기기에 푸시 알림 도착 → 탭 시 해당 화면 이동 확인.

**Acceptance Scenarios**:

1. **Given** 알림 수신에 동의한 직원의 기기에서, **When** 새 상담 기록이 저장되면, **Then** 5초 이내에 기기 푸시 알림이 도착한다.
2. **Given** 알림이 도착했을 때, **When** 사용자가 알림을 탭하면, **Then** 해당 환자의 상담 기록 화면으로 바로 이동한다.
3. **Given** 사용자가 알림 수신을 거부하면, **When** 앱을 사용할 때, **Then** 앱은 정상 동작하며 알림 없이 계속 사용할 수 있다.
4. **Given** 사용자가 알림 구독을 앱 설정에서 해제하면, **When** 이후 이벤트가 발생하면, **Then** 해당 기기로 알림이 발송되지 않는다.

---

### User Story 4 - 헤더 고정 및 새로고침 버튼 (Priority: P4)

어떤 화면에서도 상단 헤더가 항상 보인다. 헤더에는 새로고침 버튼이 있어 탭 한 번으로 현재 화면의 최신 데이터를 불러올 수 있다.

**Why this priority**: 모바일 PWA에서 주소 표시줄이 없으므로 새로고침 수단을 앱 내부에 제공해야 한다.

**Independent Test**: 긴 목록 화면에서 스크롤 후 헤더 고정 확인, 새로고침 버튼 탭 후 최신 데이터 반영 확인.

**Acceptance Scenarios**:

1. **Given** 긴 페이지에서 아래로 스크롤하면, **When** 화면이 이동해도, **Then** 상단 헤더는 항상 화면 최상단에 고정된다.
2. **Given** 헤더의 새로고침 버튼을 탭하면, **When** 데이터를 다시 불러오는 동안, **Then** 로딩 표시가 나타나고 완료 후 최신 데이터가 표시된다.

---

### User Story 5 - 푸터 브랜딩 (Priority: P5)

모든 페이지 하단에 "SUWANT holdings Inc." 브랜딩이 표시된다.

**Why this priority**: 법인 브랜딩 요구사항.

**Independent Test**: 모든 주요 페이지에서 푸터 표시 확인.

**Acceptance Scenarios**:

1. **Given** 앱의 어느 페이지에서나, **When** 페이지 하단까지 스크롤하면, **Then** "SUWANT holdings Inc." 텍스트가 포함된 푸터가 표시된다.

---

### Edge Cases

- 푸시 알림 권한을 이전에 거부한 사용자가 다시 허용하려는 경우 → 시스템 설정으로 안내
- 알림 구독 중 기기 교체 시 → 새 기기에서 재구독 필요 (기존 구독은 자동 만료)
- 오프라인 상태에서 앱 실행 시 → 캐시된 화면 표시 후 오프라인 안내
- 여러 탭/기기에서 동일 계정 사용 시 → 각 기기별 독립 알림 구독

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 앱은 PWA 설치 기준(manifest.json, HTTPS, Service Worker)을 충족하여 홈 화면 추가가 가능해야 한다.
- **FR-002**: 홈 화면 아이콘, 스플래시 화면, 앱 이름(Carelog)이 표시되어야 한다.
- **FR-003**: 앱 실행 시 세션이 유효하면 로그인 화면을 거치지 않고 메인 화면으로 이동해야 한다.
- **FR-004**: 세션 만료가 임박하면 백그라운드에서 자동으로 갱신해야 한다.
- **FR-005**: 세션 갱신 실패 시 안내 메시지와 함께 로그인 화면으로 이동해야 한다.
- **FR-006**: 앱은 사용자에게 알림 수신 동의를 요청하는 UI를 제공해야 한다.
- **FR-007**: 동의한 사용자의 기기를 알림 수신 대상으로 등록·관리해야 한다.
- **FR-008**: 새 상담 기록 저장 시 해당 기관의 모든 등록된 기기로 푸시 알림을 발송해야 한다.
- **FR-009**: 활동 피드 이벤트(consultation.created 등) 발생 시 푸시 알림을 발송해야 한다.
- **FR-010**: 푸시 알림에는 환자 이름, 이벤트 유형, 간략한 내용 미리보기가 포함되어야 한다.
- **FR-011**: 알림 탭 시 해당 환자의 상담 기록 화면으로 이동해야 한다.
- **FR-012**: 사용자가 앱 내에서 알림 구독을 해제할 수 있어야 한다.
- **FR-013**: 헤더는 모든 페이지에서 화면 상단에 항상 고정되어야 한다.
- **FR-014**: 헤더에 새로고침 버튼이 있어야 하며, 탭 시 현재 페이지 데이터를 갱신해야 한다.
- **FR-015**: 모든 페이지 하단에 "SUWANT holdings Inc." 브랜딩 푸터가 표시되어야 한다.

### Key Entities

- **알림 구독(Push Subscription)**: 기기별 알림 수신 엔드포인트, 기관 ID, 사용자 ID, 구독 상태, 생성일
- **푸시 알림(Push Notification)**: 제목, 본문, 이동 URL, 발송 대상 기관, 이벤트 유형, 발송 시각

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 모바일 기기에서 홈 화면 추가 완료까지 30초 이내 수행 가능
- **SC-002**: 앱 재실행 시 로그인 화면 없이 메인 화면 표시 성공률 99% 이상 (세션 유효 상태 기준)
- **SC-003**: 상담 기록 저장 후 5초 이내에 등록된 기기 전체에 푸시 알림 도달
- **SC-004**: 알림 수신 동의 UI에서 구독 완료까지 3단계 이내
- **SC-005**: 알림 탭 후 해당 기록 화면 이동 성공률 100%
- **SC-006**: 스크롤 중 헤더 고정 지연 없음 (1프레임 이내 응답)
- **SC-007**: 새로고침 버튼 탭 후 3초 이내 최신 데이터 반영
- **SC-008**: 모든 주요 페이지(홈, 환자 목록, 환자 상세, 설정)에서 푸터 표시

## Assumptions

- HTTPS 환경(Vercel 배포)에서만 PWA 기능(Service Worker, 푸시)이 동작한다.
- 알림 권한은 최초 1회 요청하며, 이후 앱 설정에서 관리한다.
- 푸시 알림은 기관 단위로 발송되며, 개인별 알림 설정은 v1 범위 밖이다.
- iOS Safari의 Web Push 지원(iOS 16.4+)이 필요하며, 구형 iOS에서는 알림이 동작하지 않을 수 있다.
- 세션 자동 갱신은 Supabase Auth의 기존 갱신 메커니즘을 활용한다.
- 푸터는 고정(sticky)이 아닌 페이지 하단 정적 영역으로 표시한다.
- 아이콘 디자인은 Carelog 브랜드 색상(하늘색 계열) 기반으로 자동 생성한다.
