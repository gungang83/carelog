# Feature Specification: 긴 상담 청크 분할 전사 모드 (Long Consultation Chunked Transcription)

**Feature Branch**: `010-chunked-transcription`
**Created**: 2026-06-28
**Status**: Draft
**Input**: User description: "긴 상담 청크 분할 전사 모드 — 실험실 픽커에 '긴 상담(청크)' 모드를 추가해 통짜 방식과 A/B 비교. 분할 녹음 → 조각별 전사(실패 격리) → 전체 텍스트 통짜 요약. C안(수동 모드 먼저 검증 → 나중에 자동 전환으로 승격)."

> 용어 규칙(의료법): 본 문서의 산출물은 **"상담 기록 / 전사"** 이다. "진료기록/의무기록"으로 칭하지 않는다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 긴 상담도 끊김 없이 기록된다 (Priority: P1)

상담실장이 30~40분 이상 이어지는 긴 상담(예: 외국인 신환 라미네이트 풀상담)을 녹음한 뒤 "긴 상담" 모드로 전사하면, 도중에 페이지가 죽거나 빈 화면으로 끝나지 않고 전사·요약까지 완료되어 본문에 채워지고 저장할 수 있다.

**Why this priority**: 이 기능의 존재 이유. 현재 통짜 방식은 길이 천장(타임아웃·용량·메모리)에 부딪혀 긴 상담을 아예 잃는다(서범기 18분 크래시). 긴 상담이 살아남는 것이 최소 가치다.

**Independent Test**: 실험실 워크스페이스에서 "긴 상담" 모드로 30분 이상 녹음 → 종료 → 전사문·요약이 본문에 삽입되고 저장되는지 확인하면 단독 검증 가능.

**Acceptance Scenarios**:

1. **Given** 실험실 사용자가 "긴 상담" 모드를 선택하고 35분을 녹음했을 때, **When** 녹음을 종료하면, **Then** 전 구간의 전사문이 순서대로 이어진 본문과 전체 요약이 에디터에 채워진다.
2. **Given** 통짜 방식으로는 실패하던 길이(예: 40분), **When** "긴 상담" 모드로 전사하면, **Then** "페이지를 불러올 수 없음"·전손 없이 완료된다.

---

### User Story 2 - 중간에 일부가 실패해도 전손이 아니다 (Priority: P1)

긴 상담 전사 도중 한 구간의 전사가 실패하더라도(네트워크·일시 오류 등), 나머지 구간의 전사는 보존되어 사용자가 부분 기록이라도 받는다. 실패한 구간은 재시도되거나 어디가 비었는지 명확히 표시된다.

**Why this priority**: 통짜 방식의 가장 큰 약점이 "한 번 끊기면 전부 날아감"이다. 실패 격리가 청크의 핵심 이점이며, 긴 상담일수록 일부 실패 확률이 높아 P1.

**Independent Test**: 한 구간 전사를 강제로 실패시켰을 때, 나머지 구간 전사문이 살아 본문에 들어오고 실패 구간이 표시/재시도되는지 확인.

**Acceptance Scenarios**:

1. **Given** 6개 구간 중 1개 전사가 실패했을 때, **When** 전사가 끝나면, **Then** 성공한 5개 구간 전사문은 보존되고 실패 구간은 명확히 표시(또는 자동 재시도 후 복구)된다.
2. **Given** 모든 구간 전사가 실패했을 때, **When** 전사가 끝나면, **Then** 그때만 전체 실패로 처리되고 녹음 원본은 유실되지 않는다(복구 가능).

---

### User Story 3 - 통짜 vs 청크를 직접 비교한다 (Priority: P2)

실험실 사용자가 같은 상담 상황에서 기본(통짜) 모드와 "긴 상담"(청크) 모드의 결과를 만들어 전사 정확도·요약 맥락·체감 속도를 나란히 평가할 수 있다. (C안의 목적 — 검증 후 자동 전환으로 승격 결정의 근거.)

**Why this priority**: 이번 단계는 "수동 모드로 먼저 검증"이 목적. 비교 가능성이 없으면 승격 판단 근거가 없다. 단, 긴 상담 완료(US1)·실패 격리(US2)가 먼저 동작해야 비교가 의미 있어 P2.

**Independent Test**: 같은(또는 유사) 녹음을 두 모드로 각각 전사해 결과를 사람이 비교·평가할 수 있는지 확인.

**Acceptance Scenarios**:

1. **Given** 실험실 픽커에 "긴 상담" 모드가 추가되어 있을 때, **When** 사용자가 모드를 바꿔 전사하면, **Then** 어떤 모드로 만든 결과인지 식별되고(저장 시 사용 엔진 기록), 두 결과를 비교·평가할 수 있다.

---

### User Story 4 - 긴 전사 중 진행 상태가 보인다 (Priority: P3)

긴 상담 전사는 시간이 걸리므로, 사용자에게 "처리한 구간/전체 구간"같은 진행 상태가 보여 "멈춘 것 아닌가"하는 불안과 깜깜이 대기를 줄인다.

**Why this priority**: 체감 개선이며 핵심 동작(US1·US2) 없이는 의미가 없어 P3. 없어도 기능은 성립.

**Independent Test**: 여러 구간 전사 동안 진행 표시가 갱신되는지 확인.

**Acceptance Scenarios**:

1. **Given** 긴 상담 전사가 진행 중일 때, **When** 구간들이 차례로 처리되면, **Then** 진행 상태(예: "3/8 구간 전사 중")가 갱신되어 표시된다.

---

### Edge Cases

- **짧은 녹음을 긴 상담 모드로**: 분할 간격보다 짧으면 구간이 1개 → 통짜와 사실상 동일하게 정상 동작해야 한다.
- **구간 경계에서 단어 잘림**: 고정 시간 분할 시 경계 발화가 잘릴 수 있음 → 경계 손상이 요약 품질을 크게 해치지 않아야 하며, 완화책(겹침/침묵 분할)은 점진 개선 대상.
- **녹음 중 크래시/탭 종료**: 분할 녹음 도중 죽어도 그때까지의 구간이 복구 안전망으로 살아남아야 한다.
- **빈/무음 구간**: 특정 구간이 무음이면 그 구간만 비고 나머지는 정상 처리.
- **모바일 백그라운드/화면잠금**: 기존 iOS 백그라운드 정지 제약은 이 기능 범위 밖(별도 알려진 이슈)이나, 분할이 그 위험을 키우지 않아야 한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 실험실(lab) 워크스페이스 사용자에게 "긴 상담" 전사 모드를 선택지로 제공해야 한다.
- **FR-002**: 비-실험실 워크스페이스에서는 이 모드가 적용되지 않고 기본 모드로 동작해야 한다(기존 lab 게이트 유지 — 사고 차단).
- **FR-003**: 긴 상담 모드는 한 녹음을 시간순 구간으로 나눠 각 구간을 독립적으로 전사할 수 있어야 한다.
- **FR-004**: 각 구간은 그 자체로 전사 가능한(유효한) 오디오여야 한다.
- **FR-005**: 시스템은 모든 구간의 전사 결과를 녹음 순서대로 이어붙여 하나의 전사문을 구성해야 한다.
- **FR-006**: 요약은 이어붙인 전체 전사문에 대해 1회 수행되어 전체 맥락을 반영해야 한다(구간별 개별 요약 후 병합 방식이 아니어야 한다).
- **FR-007**: 한 구간의 전사가 실패해도 나머지 구간의 전사 결과는 보존되어야 하며, 실패 구간은 자동 재시도하거나 결과에 명확히 표시되어야 한다.
- **FR-008**: 전체 구간이 모두 실패한 경우에만 전체 실패로 처리한다.
- **FR-009**: 긴 상담 모드 결과도 기존 흐름대로 본문 에디터에 삽입되고, 저장 시 어떤 모드/엔진으로 만들었는지 기록되어야 한다.
- **FR-010**: 긴 상담 모드는 음성 원본 보관(spec 009) 정책과 충돌 없이 동작해야 한다(여러 구간의 보관 방식은 보관 스키마를 깨지 않아야 한다).
- **FR-011**: 긴 상담 모드는 유실 복구 안전망(작성 중 임시 영속화)과 호환되어, 분할 녹음 도중/직후 크래시 시에도 녹음물이 복구 가능해야 한다.
- **FR-012**: 사용자는 같은 상담에 대해 긴 상담 모드와 기본 모드 결과를 비교·평가할 수 있어야 한다.
- **FR-013**: 긴 상담 전사 진행 중 사용자에게 진행 상태(예: 처리 구간/전체 구간)를 표시해야 한다.
- **FR-014**: 분할 간격보다 짧은 녹음은 단일 구간으로 처리되어 기본 모드와 동등하게 정상 동작해야 한다.

### Key Entities *(include if feature involves data)*

- **녹음 구간(Audio Segment)**: 한 녹음 세션을 시간순으로 나눈 독립 오디오 조각. 속성: 순서 인덱스, (대략) 길이, 전사 상태(성공/실패/재시도), 구간 전사문.
- **전사 결과(Transcription Run)**: 한 녹음에 대한 전사 산출물. 속성: 사용 모드/엔진, 구간 목록, 이어붙인 전체 전사문, 전체 요약, 본문 삽입 텍스트, (선택) 실패 구간 표시.
- **상담 기록(Consultation)**: 저장 시 사용 엔진(전사 모드) 식별값과 음성 보관 정보를 가진다(기존 엔티티 재사용).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 통짜 방식으로는 실패하던 길이(예: 40분) 녹음이 긴 상담 모드에서 전사·요약까지 완료되어 저장된다.
- **SC-002**: 전사 중 일부 구간 실패가 전체 전손(0건 보존)으로 이어지지 않는다 — 최소한 성공 구간의 전사문은 항상 보존된다.
- **SC-003**: 실험실 사용자가 같은 상담 상황에서 통짜와 청크 결과를 나란히 비교·평가할 수 있다.
- **SC-004**: 청크 요약의 전체 맥락 품질이 통짜 대비 동등 이상으로 평가된다(검증 목적의 사람 평가).
- **SC-005**: 긴 전사 동안 사용자가 진행 상태를 확인할 수 있어 "응답 없음/멈춤" 체감이 사라진다.
- **SC-006**: 비-실험실 워크스페이스의 전사 동작·결과는 이 기능 도입 전과 동일하다(회귀 0).

## Assumptions

- **분할 간격 기본값**: 한 구간 길이는 기본 5분 수준으로 가정한다(타임아웃·용량 여유 충분). 정확한 값은 설계 단계에서 조정 가능.
- **경계 손상 완화는 점진**: 1차는 고정 시간 분할(경계 미세 손상 허용), 겹침/침묵 기반 분할은 후속 개선으로 둔다.
- **음성 보관 방식(설계에서 확정 필요)**: spec 009는 상담당 단일 음성 경로를 쓴다. 기본 가정은 "보관 시 구간을 단일 음성으로 합치거나 대표 보관해 기존 보관 스키마를 유지"이며, 다중 구간 보관이 필요하면 별도 결정. → /plan에서 확정.
- **비교 방식**: 기존 "비교" 모드처럼 한 녹음에서 두 방식을 동시 실행하기보다, 모드를 바꿔 각각 전사하는 방식으로 비교한다고 가정(분할 녹음과 통짜 녹음 구조가 달라 동시 실행은 비범위).
- **인프라 불변**: 새 외부 서비스 없이 기존 음성 전사(Whisper)·요약(Claude) 인프라를 그대로 사용한다.
- **재사용**: 기존 엔진 레지스트리·픽커·lab 게이트 구조를 재사용한다(새 모드 추가는 그 패턴을 따른다).

## Out of Scope

- 실시간 동시통역(스트리밍) — 별건(o1 Level 2).
- 자동 전환(짧으면 통짜/길면 청크) — 이 단계는 수동 모드 검증까지. 검증 후 다음 단계에서 승격.
- iOS 백그라운드 녹음 정지 등 기존 알려진 녹음 신뢰성 이슈 — 별도 트랙.
- 구간별 개별 요약/병합(map-reduce) — 본 기능은 전체 통짜 요약만.
