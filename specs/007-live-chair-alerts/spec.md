# Feature Specification: 실시간 체어 상담기록 알림 (실시간 알림·소통 기반)

**Feature Branch**: `claude/festive-planck-FCghV` (작업 브랜치) · 스펙 디렉터리 `specs/007-live-chair-alerts`
**Created**: 2026-06-14
**Status**: Draft
**Input**: 대표님 요청 — 예미안치과처럼 체어마다 PWA를 띄워놓고 쓰는 환경에서, 한 체어(예: 2번 체어)에서 상담 기록이 올라오면 매장 내 다른 모든 화면에 실시간으로 알림(팝업+소리)과 푸시가 오게. 앞으로 알림·소통 기능이 계속 보완될 것이므로 재사용 가능한 기반으로.

---

## 핵심 가치 (한 줄)

매장 안 어느 체어에서 상담 기록이 올라오든, **열려 있는 모든 직원 화면이 즉시(보고 있지 않아도 소리로) 알아차리게** 한다 — 기록 누락·확인 지연 0. 이번 구현은 향후 알림·소통 기능의 **공용 파이프라인** 첫 도입이다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — 실시간 인앱 알림 + 목록 자동 갱신 (Priority: P1)

데스크 직원이 홈 화면을 띄워둔 상태에서, 2번 체어의 원장이 상담을 녹음·저장한다. 데스크 화면은 **새로고침 없이** 곧바로 화면 상단(또는 모서리)에 "2번 체어 · 상담 기록 도착" 팝업/토스트가 뜨고, 홈의 "미연결 기록" 목록에 새 카드가 즉시 나타난다.

**Why this priority**: 이게 MVP. 항상 띄워두는 화면에 "방금 올라온 기록"이 즉시 보이는 것만으로 현장의 핵심 문제(다른 체어 기록을 모르고 지나침)를 해결한다. 소리·푸시 없이도 단독으로 가치가 있다.

**Independent Test**: 두 개의 브라우저(같은 기관 로그인)를 열고, 한쪽에서 체어 기록을 저장 → 다른 쪽 화면에 5초 내 토스트가 뜨고 목록이 수동 새로고침 없이 갱신되는지 확인.

**Acceptance Scenarios**:

1. **Given** 같은 기관 직원이 다른 기기에서 홈을 열어둠, **When** 2번 체어에서 새 상담 기록이 저장됨, **Then** 그 기기 화면에 "2번 체어 · 상담 기록 도착" 토스트가 뜨고 미연결 기록 목록에 새 항목이 자동 추가된다.
2. **Given** 기록을 방금 저장한 바로 그 기기, **When** 저장이 완료됨, **Then** 자기 자신에겐 중복 팝업이 뜨지 않는다(에코 방지).
3. **Given** 다른 기관(B병원) 직원 화면, **When** A병원 2번 체어에서 기록이 올라옴, **Then** B병원 화면엔 아무 알림도 오지 않는다(기관 격리).
4. **Given** 화면이 잠시 네트워크가 끊겼다 다시 연결됨, **When** 끊긴 동안 올라온 기록이 있음, **Then** 재연결 시 목록이 최신 상태로 맞춰진다(놓친 항목 반영).

---

### User Story 2 — 소리 안내 (Priority: P2)

직원이 화면을 직접 보고 있지 않아도(다른 환자 응대 중 등) **소리**로 새 기록 도착을 알아챈다. 화면에 처음 한 번 "🔔 알림 소리 켜기"를 누르면 그날 그 화면에서는 이후 도착마다 효과음이 울린다.

**Why this priority**: 시각 알림만으로는 안 보고 있을 때 놓친다. 소리가 현장 체감 효과가 큼. 단, 브라우저 정책상 1회 활성화가 필요해 P1과 분리.

**Independent Test**: 화면에서 "알림 소리 켜기"를 누른 뒤 다른 기기에서 기록을 저장 → 효과음이 재생되는지 확인. 활성화 전에는 소리가 나지 않고 시각 알림만 뜨는지 확인.

**Acceptance Scenarios**:

1. **Given** "알림 소리 켜기"를 1회 누른 화면, **When** 새 체어 기록이 도착, **Then** 효과음이 재생된다.
2. **Given** 소리를 한 번도 활성화하지 않은 화면, **When** 새 기록이 도착, **Then** 소리는 나지 않고 시각 알림만 표시된다(오류 없이).
3. **Given** 짧은 시간에 여러 기록이 연달아 도착, **When** 알림이 발생, **Then** 소리가 겹쳐 시끄럽지 않도록 디바운스(짧은 간격 묶음)된다.
4. **Given** 직원이 알림 소리를 끔(설정), **When** 새 기록이 도착, **Then** 시각 알림은 유지되고 소리만 나지 않는다.

---

### User Story 3 — 화면 꺼짐/백그라운드 대비 푸시 알림 (Priority: P3)

직원 기기 화면이 꺼져 있거나 Carelog가 백그라운드일 때도, 새 체어 상담 기록이 올라오면 **OS 푸시 알림**이 온다(기존 푸시 알림 기능 확장 — 현재는 환자 상담 저장 때만 발송됨).

**Why this priority**: 항상 띄워두는 화면(P1·P2)으로 대부분 커버되지만, 화면이 꺼진 보조 기기/개인 휴대폰까지 닿게 하는 보완층. 인앱 알림보다 후순위.

**Independent Test**: 기기에서 푸시 알림을 허용하고 Carelog를 백그라운드로 둔 뒤 다른 기기에서 체어 기록 저장 → OS 알림이 오는지 확인.

**Acceptance Scenarios**:

1. **Given** 푸시 알림을 허용하고 백그라운드 상태인 직원 기기, **When** 새 체어 기록이 올라옴, **Then** "새 상담 기록 도착" OS 푸시 알림이 표시되고, 누르면 해당 화면(미연결 기록)으로 이동한다.
2. **Given** 화면을 띄워둔(포그라운드) 기기, **When** 같은 기록 이벤트, **Then** 인앱 알림과 OS 푸시가 **중복으로 시끄럽지 않게** 처리된다(포그라운드는 인앱 우선).

---

### Edge Cases

- **에코 방지**: 기록을 생성한 본인 화면엔 팝업/소리를 띄우지 않는다(목록 갱신은 함).
- **연속/중복 이벤트**: 짧은 간격 다발 도착 시 소리·팝업 디바운스(묶음 표시).
- **소리 자동재생 차단**: 1회 활성화 전에는 소리 미재생, 시각 알림만. 활성화 상태는 화면 단위로 유지.
- **오프라인→재연결**: 끊긴 동안 이벤트는 재연결 시 목록 재동기화로 보정(실시간 팝업은 놓칠 수 있음 — 푸시가 보완).
- **기관 격리**: 알림은 같은 institution 직원 화면에만. 타 기관·환자 화면엔 전파 0.
- **권한 거부**: 푸시 권한이 없으면 인앱 알림만 동작(오류 없이 graceful).
- **다중 화면**: 한 직원이 여러 기기를 띄워도 각 화면이 독립적으로 알림.
- **이미 연결된 기록**: 알림 트리거는 "새 체어 상담 기록 생성(업로드)" 시점. 환자 연결/편집은 이번 범위의 알림 대상 아님(향후 확장).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 같은 기관(institution) 안에서 **새 체어 상담 기록이 생성(업로드)**되면, 그 기관 소속으로 화면을 열어둔 모든 직원 클라이언트에 **실시간 이벤트를 전파**해야 한다.
- **FR-002**: 이벤트를 받은 화면은 **수동 새로고침 없이** 관련 목록(홈 미연결 기록 등)을 갱신해야 한다.
- **FR-003**: 이벤트를 받은 화면은 **시각 알림(팝업/토스트)**을 표시해야 하며, 알림에는 **출처 체어 식별자(예: "2번 체어")**와 도착 사실이 포함되어야 한다.
- **FR-004**: 시스템은 기록을 **생성한 본인 화면에는 팝업/소리 알림을 띄우지 않아야** 한다(목록 갱신은 허용).
- **FR-005**: 직원은 화면에서 **알림 소리를 1회 활성화**할 수 있어야 하며, 활성화 후 새 기록 도착 시 **효과음**이 재생되어야 한다.
- **FR-006**: 직원은 **알림 소리를 켜고/끌 수** 있어야 한다(기본값은 합리적 기본 — 활성화 전 무음).
- **FR-007**: 시스템은 짧은 간격에 다발로 도착하는 이벤트의 **소리·팝업을 디바운스**해야 한다(과도한 중복 방지).
- **FR-008**: 알림은 **같은 기관 직원에게만** 전파되어야 하며, 타 기관·환자 화면에는 전파되지 않아야 한다.
- **FR-009**: 시스템은 화면이 **꺼짐/백그라운드**일 때 새 체어 기록 도착에 대해 **OS 푸시 알림**을 발송해야 한다(기존 직원 푸시 알림 기능을 체어 기록 이벤트로 확장).
- **FR-010**: 포그라운드(화면 켜짐) 상태에서는 인앱 알림과 OS 푸시가 **중복으로 과하게 표시되지 않도록** 조정되어야 한다.
- **FR-011**: 화면이 일시적으로 연결이 끊겼다 **재연결**되면 목록이 최신 상태로 **재동기화**되어야 한다.
- **FR-012**: 알림 전파 메커니즘은 **체어 기록 외 다른 기관 내 이벤트로 재사용 가능**하도록 일반화된 형태로 설계되어야 한다(향후 직원 호출·환자 도착·상담 배정 등). *비고: 일반화 수준의 구체 범위는 plan 단계에서 확정.*

### Key Entities *(데이터 포함)*

- **기관 내 알림 이벤트(Notification event)**: 어떤 일이 일어났는지(예: 새 체어 기록 도착), 출처(체어/기록 식별자), 발생 시각, 대상 기관. 향후 다른 이벤트 유형으로 확장.
- **체어 상담 기록(Chair record)**: 알림의 원천. 출처 체어 번호·기관·생성자(직원)·생성 시각을 가짐(기존 자산).
- **직원 화면 세션(Subscriber)**: 알림을 받는 열린 클라이언트. 소속 기관, 알림 소리 활성화 여부, 본인 식별(에코 방지용).
- **알림 환경설정(Notification preference)**: 소리 on/off, (확장) 알림 종류별 on/off.
- **푸시 구독(Push subscription)**: 화면 꺼짐 대비 OS 푸시 발송 대상(기존 자산).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 한 체어에서 상담 기록이 올라온 뒤 **5초 이내**에 같은 기관의 열린 모든 화면에 알림이 표시되고 목록이 갱신된다.
- **SC-002**: 알림 소리를 1회 활성화한 화면은 **그날의 이후 모든 도착**에서 추가 조작 없이 소리가 울린다.
- **SC-003**: 기록을 올린 본인 화면에는 **중복 팝업/소리가 0건** 발생한다.
- **SC-004**: 타 기관 화면으로의 알림 오발송 **0건**(기관 격리 100%).
- **SC-005**: 화면을 꺼둔(또는 백그라운드) 기기에서 푸시 허용 시, 새 체어 기록 도착에 대해 OS 푸시가 도착한다.
- **SC-006**: 현장 검증 — "다른 체어 기록을 모르고 지나치는" 사례가 도입 전 대비 체감상 사라진다(예미안 1주 운영 피드백 기준).

## Assumptions

- **사용 기기/플랫폼 (확정, 2026-06-14)**: **PC/노트북 브라우저 + 안드로이드 태블릿(PWA) + 직원 개인 안드로이드 휴대폰(보조)**. 전부 Chromium 계열로 **실시간·소리·Web Push 모두 제약 없이 지원**된다. **애플(iPad/iPhone·iOS)은 초기 타깃에 없음 → v1 범위 밖**(향후 필요 시 iOS PWA 푸시/소리 제약을 별도 검토).
- **소리 방식**: 1차는 **짧은 효과음(띵동)**을 기본으로 가정한다. 한국어 음성 안내(TTS, "2번 체어 상담 기록이 올라왔습니다")는 옵션으로 후속 검토(범위 확장 가능).
- **트리거 시점**: 알림 트리거는 "새 체어 상담 기록 생성(업로드)" 1개 이벤트로 한정(이번 범위). 환자 연결·편집·삭제 알림은 향후 확장.
- **인앱 실시간은 화면이 열려 있을 때만** 동작하며, 닫힌 화면은 푸시(US3)가 보완한다(두 층 상호보완) — 예미안의 "항상 띄워둠" 사용패턴과 부합.
- **읽음 표시·알림 센터(인박스)·직원 간 메시지**는 이번 범위 밖(향후 알림·소통 축에서 별도 스펙).
- **재사용 기반**: 기존 직원 푸시 알림 인프라를 재활용하고, 실시간 전파 계층을 신규 도입하되 향후 이벤트 유형 확장이 쉬운 형태로 둔다(구체 일반화는 plan).
- **권한·보안**: 알림 데이터에는 환자 식별정보·진료 본문을 싣지 않는다(체어 번호·도착 사실 등 최소 정보만). 기관 격리는 기존 다중 테넌트 규칙을 따른다.
