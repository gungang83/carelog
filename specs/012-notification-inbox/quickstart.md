# Quickstart: 알림함 검증

> 대상: 예미안(또는 임의 기관) 직원 계정. 사전: 마이그레이션 적용 + `notifications` realtime publication 추가.

## 배포 전 필수
1. 마이그레이션(`notifications`·`notification_reads`·RLS) 적용.
2. `alter publication supabase_realtime add table public.notifications;` 적용(마이그레이션에 포함).
3. `npm run build` 그린.

## 시나리오 1 — 알림 적재·배지 (SC-001)
1. 직원 A 로그인(홈).
2. (같은 기관) 새 상담 기록 저장(체어 즉시기록/상담보드).
3. **기대**: 헤더 종에 미읽음 배지(+1)가 (실시간) 뜨고, 종을 누르면 "새 상담 기록…" 알림이 최신순 최상단.

## 시나리오 2 — 읽음 관리 (SC-002)
1. 알림 클릭 → **기대**: 읽음 처리(배지 -1) + 링크(/records 등) 이동.
2. 드롭다운에서 읽은 알림의 ✓ 클릭 → **기대**: 안읽음 복귀(배지 +1).
3. '전체 읽음' → **기대**: 모든 미읽음 읽음, 배지 사라짐.

## 시나리오 3 — 실시간 (SC-001/US3)
1. A는 알림함을 띄워둠. 다른 기기/직원 B가 상담 저장.
2. **기대**: 새로고침 없이 수초 내 A의 목록·배지 갱신.

## 시나리오 4 — 격리·대상·본인읽음 (SC-003/004)
1. 다른 기관 알림은 A에게 안 보임.
2. 'admins' 대상 알림은 staff 역할에게 안 보임(admin에게만).
3. A가 읽은 알림이 B에겐 여전히 미읽음.

## 시나리오 5 — PWA 배지 (FR-009)
1. PWA 설치 기기에서 미읽음 존재 시 홈 아이콘 배지 표시(지원 기기).

## 시나리오 6 — 회귀 (SC-005)
- 기존 헤더(새로고침·소리)·실시간 토스트(LiveAlertsProvider) 정상.
- 상담 저장/푸시 정상(알림 적재 실패해도 저장·푸시 비차단).
