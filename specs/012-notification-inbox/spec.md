# Feature Specification: 알림함 (Notification Inbox)

**Feature Branch**: `012-notification-inbox`
**Created**: 2026-06-28
**Status**: Draft
**Input**: EO '알림' 기능을 Carelog에 같은 UX로 포팅. 헤더 종 + 드롭다운 알림함 + 읽음 관리. EO 코드 읽기전용 교차검증 완료(NotificationBell.tsx, lib/notifications.ts, api/notifications/*, notifications+notification_reads 2테이블).

> 용어 규칙(의료법): 산출물은 "상담 기록". "진료기록/의무기록" 미사용.

## User Scenarios & Testing

### US1 — 지나간 알림을 한곳에서 다시 본다 (Priority: P1)
직원이 헤더의 종 아이콘을 누르면 그동안 발생한 알림(새 상담 기록 저장 등)이 최신순 목록으로 보인다. 푸시가 와도 못 봤거나 사라진 알림을 여기서 다시 확인한다.
**Acceptance**:
1. Given 직원 대상 이벤트가 여러 번 발생했을 때, When 종을 누르면, Then 알림이 최신순으로(제목·본문·상대시간·타입 아이콘) 나열된다.
2. Given 알림이 하나도 없을 때, When 종을 누르면, Then "알림이 없습니다" 안내가 보인다.

### US2 — 읽음/안읽음 관리 (Priority: P1)
미읽음 개수가 종에 배지로 표시되고, 알림을 읽으면 배지가 줄고, 개별 읽음/읽음취소·전체 읽음이 가능하다.
**Acceptance**:
1. Given 미읽음 알림이 N개일 때, When 헤더를 보면, Then 종에 N(9 초과 시 9+) 배지가 보인다.
2. When 알림을 클릭하면, Then 읽음 처리되고 해당 화면으로 이동하며 배지가 1 줄어든다.
3. When '전체 읽음'을 누르면, Then 모든 미읽음이 읽음 처리되고 배지가 사라진다.
4. When 읽은 알림의 읽음 표시를 다시 누르면, Then 안읽음으로 되돌아간다.

### US3 — 새 알림 실시간 반영 (Priority: P2)
다른 직원이 상담 기록을 저장하는 등 이벤트가 생기면, 새로고침 없이 알림함과 배지에 즉시 반영된다.
**Acceptance**:
1. Given 알림함을 띄워둔 상태에서, When 같은 기관에 새 알림이 생기면, Then 수초 내 목록·배지가 갱신된다.

### US4 — 기관 격리·대상 필터·본인 읽음상태 (Priority: P1)
알림은 소속 기관 것만 보이고, 대상(전체/관리자/특정인)에 맞는 사람에게만 보이며, 읽음 상태는 사람마다 독립이다.
**Acceptance**:
1. Given 다른 기관의 알림, When 내 알림함을 보면, Then 보이지 않는다.
2. Given '관리자 대상' 알림, When 일반 직원이 보면, Then 보이지 않는다.
3. Given 한 알림을 A가 읽었을 때, When B가 보면, Then B에게는 여전히 미읽음이다.

### Edge Cases
- PWA로 설치된 경우 홈 아이콘에도 미읽음 배지가 표시된다(지원 기기).
- 실시간 연결이 끊겨도 주기적 갱신으로 누락 없이 따라잡는다.
- 같은 알림에 대한 중복 읽음 처리가 오류를 일으키지 않는다(멱등).
- 기존(이 기능 도입 전)에 이미 발송돼 적재되지 않은 푸시는 소급 표시되지 않을 수 있다(가는 시점부터 누적). 초기 화면이 비지 않도록 과거 이벤트 시드는 선택.

## Requirements

- **FR-001**: 시스템은 직원 대상 이벤트(예: 새 상담 기록 저장, 상담 환자 연결)가 발생하면 알림을 **기록**하고 동시에 기존 푸시를 발송해야 한다(한 흐름).
- **FR-002**: 알림은 **제목·본문·타입·이동 링크·생성시각·대상(recipients)** 을 가진다. 대상은 전체/관리자/특정인을 표현할 수 있어야 한다.
- **FR-003**: 직원은 헤더의 종에서 자신이 받을 알림을 **최신순**으로 열람할 수 있어야 한다(최근 N건).
- **FR-004**: 미읽음 개수가 종에 **배지**로 표시되어야 한다(상한 표기 9+).
- **FR-005**: 알림 클릭 시 **읽음 처리 + 연결 화면 이동**, 개별 **읽음/읽음취소**, **전체 읽음**이 가능해야 한다.
- **FR-006**: 읽음 상태는 **사용자별로 독립**이며 멱등해야 한다(중복 읽음/전체읽음이 오류 없음).
- **FR-007**: 알림은 **소속 기관으로 격리**되고, **대상(recipients)에 맞는 사용자에게만** 보여야 한다(관리자 전용은 일반 직원에게 비노출).
- **FR-008**: 새 알림은 **실시간으로** 반영되며, 실시간 실패 시 **주기적 갱신**으로 보완해야 한다.
- **FR-009**: 지원 환경(PWA)에서는 홈 화면 앱 아이콘에 미읽음 **배지**가 반영되어야 한다.
- **FR-010**: 알림 본문/목록에는 환자 민감정보(주민번호 등)를 평문 노출하지 않는다.
- **FR-011**: 알림 열람·읽음 처리는 서버 권위로 수행되며(직접 클라이언트 쓰기 금지) 기관 격리는 RLS로도 강제되어야 한다.

### Key Entities
- **알림(Notification)**: 한 건의 broadcast 메시지. 속성: 기관, 생성시각, 제목, 본문, 타입, 이동 링크, 대상(전체/관리자/특정인).
- **읽음 표시(Notification Read)**: 특정 사용자가 특정 알림을 읽었다는 표시(있으면 읽음, 없으면 안읽음). 사용자×알림 1:1.

## Success Criteria
- **SC-001**: 새 상담 기록 저장 등 직원 대상 이벤트 발생 시 알림함에 즉시(수초 내) 쌓이고 배지 카운트가 오른다.
- **SC-002**: 알림 클릭으로 읽음 처리·화면 이동, 읽음/안읽음 토글·전체 읽음이 모두 동작한다.
- **SC-003**: 다른 기관 알림은 안 보이고, 본인 읽음 상태만 반영된다(다른 사람의 읽음에 영향 없음).
- **SC-004**: 관리자 전용 알림이 일반 직원에게 보이지 않는다.
- **SC-005**: 기존 화면(헤더·실시간 토스트 등) 회귀 0.

## Assumptions
- 알림함은 **직원(기관 멤버) 대상**. 환자 포털 알림함은 별건(후속).
- 읽음 키는 **로그인 사용자(user_id)**. 대상 필터의 '관리자'는 `institution_members.role`(admin) 기준.
- 채널은 **웹푸시 + 인앱 알림함**까지. 이메일·SMS 채널 정책은 비범위(후속).
- 최근 N=최대 50건 노출(EO와 동일 수준), 과도 로드 방지.
- 기존 실시간 토스트(LiveAlertsProvider)는 유지 — 알림함은 '영속 기록·읽음관리'를 더한다(역할 분리).

## Out of Scope
- 환자 포털 알림함(환자 푸시 적재).
- 이메일·SMS 채널 및 알림 설정 페이지(채널 on/off).
- 과거 미적재 푸시의 완전 소급(선택적 시드만 후속 고려).
