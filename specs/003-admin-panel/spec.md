## User Scenarios & Testing *(mandatory)*

### User Story 1 — 복수 기관 전환 (Priority: P1)

복수 기관에 소속된 직원이 로그인 후 헤더 드롭다운에서 현재 작업할 기관을 선택한다. 선택한 기관 컨텍스트로 대시보드 전체가 전환되어, 해당 기관의 환자·상담 데이터만 표시된다. 단일 기관 소속 직원에게는 드롭다운이 표시되지 않는다.

**Why this priority**: 복수 기관 근무자가 기관 간 데이터를 혼동 없이 다룰 수 있는 가장 기본 기능. 이 기능 없이는 복수 기관 소속 계정이 어느 기관 데이터를 보는지 알 수 없다.

**Independent Test**: 동일 계정으로 두 기관 멤버십 부여 → 로그인 → 헤더 드롭다운 확인 → 기관 전환 → 대시보드 데이터가 선택 기관 것으로 바뀌는지 확인

**Acceptance Scenarios**:

1. **Given** 두 기관에 소속된 직원이 로그인한 상태, **When** 헤더의 기관 이름을 클릭, **Then** 소속 기관 목록이 드롭다운으로 표시된다
2. **Given** 드롭다운에서 다른 기관을 선택, **When** 선택 완료, **Then** 대시보드가 해당 기관의 환자·상담 데이터로 즉시 전환된다
3. **Given** 단일 기관 소속 직원이 로그인, **When** 헤더 확인, **Then** 드롭다운 없이 기관명만 텍스트로 표시된다
4. **Given** 기관 전환 후 새로고침, **When** 페이지 로드, **Then** 마지막으로 선택한 기관이 유지된다

### User Story 2 — 기관 직원 권한 관리 (Priority: P1)

기관 관리자(owner)가 설정 페이지에서 소속 직원 목록을 조회하고, 각 직원의 Carelog 접근 권한을 활성화/비활성화한다. 접근이 비활성화된 직원은 로그인해도 해당 기관의 데이터에 접근할 수 없다.

**Why this priority**: 퇴직·휴직 직원의 데이터 접근 차단이 의료 정보 보안의 핵심 요구사항이다. 관리자가 즉시 제어할 수 없으면 보안 사고 위험이 있다.

**Independent Test**: 기관 관리자 로그인 → 설정 → 직원 목록 확인 → 특정 직원 비활성화 → 해당 직원 계정으로 로그인 시 접근 차단 확인

**Acceptance Scenarios**:

1. **Given** 기관 관리자가 설정 페이지 접근, **When** 직원 관리 탭 진입, **Then** 소속 직원 이름·이메일·역할·상태(활성/비활성)가 목록으로 표시된다
2. **Given** 직원 목록에서 특정 직원의 상태 토글, **When** 비활성화 확인, **Then** 즉시 적용되어 해당 직원은 이 기관 데이터에 접근 불가 상태가 된다
3. **Given** 비활성화된 직원이 로그인, **When** 해당 기관 대시보드 접근 시도, **Then** "접근 권한이 없습니다" 안내 메시지가 표시된다
4. **Given** 기관 관리자가 직원 초대, **When** 초대 이메일 발송, **Then** 직원이 수락 후 목록에 추가된다

### User Story 3 — 최고 관리자 통합 관리 (Priority: P2)

최고 관리자(mobys0416@gmail.com)는 모든 기관 목록을 조회하고, 각 기관의 직원 권한을 관리하며, 기관을 활성화/비활성화할 수 있다. 일반 기관 관리자 화면과 구분된 최고 관리자 전용 대시보드에서 이루어진다.

**Why this priority**: 서비스 운영자가 전체 현황을 파악하고 문제 기관·계정을 즉시 제어할 수 있어야 한다. 단, 개별 기관 권한 관리(US2)가 먼저 동작해야 이 기능이 의미 있다.

**Independent Test**: mobys0416@gmail.com으로 로그인 → 최고 관리자 메뉴 진입 → 전체 기관 목록 확인 → 특정 기관의 직원 권한 변경 → 해당 직원 접근 상태 확인

**Acceptance Scenarios**:

1. **Given** 최고 관리자 계정으로 로그인, **When** 대시보드 접근, **Then** 일반 직원에게는 보이지 않는 "관리자" 메뉴가 헤더에 표시된다
2. **Given** 최고 관리자 패널 진입, **When** 기관 목록 조회, **Then** 전체 등록 기관의 이름·직원 수·생성일·상태가 표시된다
3. **Given** 특정 기관 선택, **When** 직원 목록 조회, **Then** 해당 기관의 직원 전체와 권한 상태를 확인·변경할 수 있다
4. **Given** 기관 비활성화 처리, **When** 해당 기관 직원 로그인 시, **Then** 기관 데이터에 접근할 수 없고 서비스 중단 안내가 표시된다

---

### User Story 4 — 기관 프로필 설정 (Priority: P3)

기관 관리자가 설정 페이지에서 기관 기본 정보(기관명, 대표 전화, 주소)를 수정하고 저장한다.

**Why this priority**: 서비스 운영에 필수적이지는 않지만, 환자에게 발송되는 SMS 등에 기관명이 표시되므로 수정 기능이 있어야 한다.

**Independent Test**: 기관 관리자 → 설정 → 기관 정보 수정 → 저장 → 환자 초대 SMS에 변경된 기관명 반영 확인

**Acceptance Scenarios**:

1. **Given** 기관 관리자가 설정 > 기관 정보 탭 진입, **When** 기관명 수정 후 저장, **Then** 헤더와 SMS 발송 내용에 새 기관명이 반영된다
2. **Given** 기관명을 빈 값으로 저장 시도, **When** 저장 버튼 클릭, **Then** "기관명은 필수입니다" 오류 메시지가 표시되고 저장되지 않는다

---

### Edge Cases

- 최고 관리자가 자기 자신의 권한을 비활성화하려 하면? → 차단하고 경고 표시
- 기관의 마지막 owner를 비활성화하려 하면? → 차단하고 "최소 1명의 관리자가 필요합니다" 안내
- 복수 기관 소속 직원이 한 기관에서 비활성화되면? → 해당 기관만 접근 불가, 다른 기관은 정상 접근
- 최고 관리자 이메일을 다른 계정이 사용하려 하면? → 이메일 기준으로만 판단하므로 계정 탈취 시 위험 — 향후 별도 권한 테이블로 이전 고려

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 복수 기관 소속 직원은 헤더 드롭다운으로 현재 기관을 전환할 수 있어야 한다
- **FR-002**: 기관 전환 선택은 세션 동안 유지되어야 하며 새로고침 후에도 유지되어야 한다
- **FR-003**: 기관 관리자는 소속 직원 목록(이름, 이메일, 역할, 활성 상태)을 조회할 수 있어야 한다
- **FR-004**: 기관 관리자는 직원의 Carelog 접근 권한을 즉시 활성화/비활성화할 수 있어야 한다
- **FR-005**: 접근이 비활성화된 직원은 해당 기관 데이터에 접근할 수 없어야 한다
- **FR-006**: 최고 관리자(mobys0416@gmail.com)는 전체 기관 목록과 각 기관 직원 현황을 조회할 수 있어야 한다
- **FR-007**: 최고 관리자는 기관 및 직원 권한을 모든 기관에 걸쳐 관리할 수 있어야 한다
- **FR-008**: 기관 관리자는 기관 기본 정보(기관명 등)를 수정할 수 있어야 한다
- **FR-009**: 기관의 마지막 관리자는 비활성화할 수 없어야 한다
- **FR-010**: 최고 관리자는 자기 자신의 접근 권한을 비활성화할 수 없어야 한다

### Key Entities

- **Institution** (기관): 치과 등 의료기관. 이름, 유형, 활성 상태 보유
- **InstitutionMember** (기관 멤버): 기관-직원 연결. 역할(owner/member), 활성 상태 보유
- **SuperAdmin** (최고 관리자): 이메일로 식별되는 특수 권한 주체. 현재는 단일 계정(mobys0416@gmail.com)

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 복수 기관 직원이 기관 전환을 3초 이내 완료할 수 있다
- **SC-002**: 기관 관리자가 직원 권한 변경 후 1분 이내 해당 직원의 접근 상태에 반영된다
- **SC-003**: 최고 관리자가 전체 기관 현황을 단일 화면에서 확인할 수 있다
- **SC-004**: 권한이 없는 직원이 데이터에 접근하려 할 때 100% 차단된다

## Assumptions

- 최고 관리자 식별은 현재 이메일 주소(mobys0416@gmail.com) 기준으로 하드코딩. 향후 별도 권한 테이블로 이전 가능
- 직원 비활성화는 계정 삭제가 아닌 기관별 접근 차단으로 처리 (복수 기관 고려)
- 알림 관리는 이번 범위에서 제외 — 추후 별도 스펙으로 분리
- 직원 초대 UI는 이미 존재하는 `inviteStaff` 백엔드를 활용하며, 프론트엔드 UI를 이번 스펙에서 포함
- 기관 설정에서 수정 가능한 정보는 기관명에 한정 (전화, 주소는 추후 확장)
- 복수 기관 전환 시 선택 기관은 쿠키로 유지 (서버 렌더링과 호환)
