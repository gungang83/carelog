# Quickstart: 환자 포털 검증 시나리오

**Date**: 2026-05-10

---

## 사전 준비

1. Solapi 계정 생성 → API 키/시크릿 발급
2. 발신 번호 등록 (Solapi 대시보드)
3. `.env.local` 및 Vercel 환경변수에 추가:
   ```
   SOLAPI_API_KEY=...
   SOLAPI_API_SECRET=...
   SOLAPI_SENDER_PHONE=01000000000
   ```
4. DB 마이그레이션 실행: `supabase/migrations/20260510000001_patient_portal.sql`

---

## 시나리오 1: 직원이 환자에게 초대 문자 발송

1. 직원 계정으로 `/login` → 로그인
2. 환자 목록에서 환자 상세 페이지 진입
3. "상담 공유" 버튼 클릭
4. 전화번호 확인 (자동 채워짐) + "개인정보 제공에 동의합니다" 체크
5. "문자 발송" 클릭
6. **예상 결과**: 해당 번호로 문자 수신, "전송 완료" 메시지 표시

---

## 시나리오 2: 신규 환자 가입 (초대 링크 경유)

1. 수신한 문자의 링크 클릭 → `/p/[token]` 진입
2. 토큰 유효성 확인 화면: 기관명 + 환자 이름 표시
3. "전화번호로 가입하기" → 전화번호 입력
4. OTP 문자 수신 → 6자리 입력
5. **예상 결과**: 계정 생성 + `/portal/records`로 이동 + 해당 치과 상담 내역 표시

---

## 시나리오 3: 기존 계정 + 새 치과 연결

1. 이미 A치과에서 가입한 환자가 B치과 초대 링크 클릭
2. 본인 전화번호 OTP 인증 완료
3. **예상 결과**: 새 계정 생성 없이 기존 계정에 B치과 상담이 추가됨, 목록에서 A+B치과 내역 모두 표시

---

## 시나리오 4: 재방문 로그인 (초대 링크 없이)

1. `/portal/login` 접속
2. 전화번호 입력 → OTP 발송 → 입력
3. **예상 결과**: 기존 계정 로그인 + 상담 내역 목록 표시

---

## 시나리오 5: 만료된 초대 링크

1. 발송 후 73시간 이상 지난 링크 클릭
2. **예상 결과**: "링크가 만료되었습니다. 치과에 재전송을 요청하세요" 안내

---

## 시나리오 6: OTP 3회 오류 → 잠금

1. `/portal/login` → 전화번호 입력 → OTP 발송
2. 잘못된 코드 3회 연속 입력
3. **예상 결과**: "잠시 후 다시 시도해 주세요 (10분)" 안내, 추가 시도 차단

---

## 시나리오 7: 환자 데이터 격리 검증

1. 환자 A로 로그인
2. `/portal/records` 응답 확인
3. **예상 결과**: 환자 A의 상담만 표시. 환자 B, C의 데이터 미노출.
