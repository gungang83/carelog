# Feature Specification: 환자 포털 — 문자 초대 · OTP 가입 · 통합 상담 조회

**Feature Branch**: `002-patient-portal`  
**Created**: 2026-05-10  
**Status**: Draft

---

## User Scenarios & Testing *(mandatory)*

### User Story 1 — 직원이 환자에게 상담 내역 공유 문자 발송 (Priority: P1)

치과 직원이 상담을 마친 후 환자의 전화번호를 입력하고 동의를 확인한 뒤 "문자 전송" 버튼을 누르면, 환자에게 케어로그 가입 링크가 포함된 SMS가 발송된다.

**Why this priority**: 이 단계가 없으면 환자가 서비스에 진입할 방법이 없다. 전체 기능의 시작점.

**Independent Test**: 직원 로그인 → 환자 상세 페이지에서 "문자 전송" 버튼 클릭 → 전화번호 입력 + 동의 체크 → 전송 → 해당 번호로 SMS 수신 확인

**Acceptance Scenarios**:

1. **Given** 직원이 환자 상세 페이지에 있고 **When** 환자 전화번호 입력 + 개인정보 제공 동의 체크 + 전송 버튼 클릭, **Then** SMS가 발송되고 "전송 완료" 메시지가 표시된다
2. **Given** 전화번호 미입력 상태에서 **When** 전송 버튼 클릭, **Then** "전화번호를 입력해 주세요" 오류 메시지가 표시되고 SMS는 발송되지 않는다
3. **Given** 동의 체크 미완료 상태에서 **When** 전송 버튼 클릭, **Then** "동의가 필요합니다" 오류 메시지가 표시되고 발송되지 않는다
4. **Given** SMS 발송 성공 이후 **When** 같은 환자에게 재발송, **Then** 이전 초대 링크는 무효화되고 새 링크로 재발송된다

### User Story 2 — 환자가 링크를 통해 케어로그에 최초 가입 (Priority: P1)

환자가 SMS의 링크를 클릭하면 케어로그 환자 가입 페이지로 이동한다. 주민등록번호와 전화번호를 입력하면 시스템이 주민번호 해시로 본인을 확인하고 OTP를 발송한다. OTP 인증 완료 시 환자 계정이 생성(또는 기존 계정 연결)되고 해당 치과의 상담 내역과 자동으로 연결된다.

**Why this priority**: 가입 흐름 완성 없이는 환자가 서비스를 이용할 수 없다. US1과 함께 MVP의 핵심.

**Independent Test**: 링크 클릭 → 주민번호 + 전화번호 입력 → 본인 확인 → OTP 인증 → 계정 생성 → 상담 내역 1건 이상 표시 확인

**Acceptance Scenarios**:

1. **Given** 유효한 초대 링크를 클릭한 신규 환자가 **When** 주민번호(앞 6자리 + 뒤 7자리) + 전화번호 입력 후 제출, **Then** 주민번호가 해당 치과의 환자 기록과 일치하면 OTP가 발송된다
2. **Given** OTP 입력 완료 시 **Then** 환자 계정이 생성되고 해당 치과 상담 내역 조회 화면으로 이동한다
3. **Given** 이미 케어로그 계정이 있는 환자(동일 주민번호 해시)가 새 치과 초대 링크를 통해 가입 시 **Then** 기존 계정에 새 치과 상담이 추가 연결된다 (계정 중복 생성 없음)
4. **Given** 입력한 주민번호가 치과의 환자 기록과 불일치할 때 **Then** "입력 정보가 일치하지 않습니다" 오류가 표시되고 OTP는 발송되지 않는다
5. **Given** 만료된 초대 링크를 클릭했을 때 **Then** "링크가 만료되었습니다. 치과에 재전송을 요청하세요" 안내가 표시된다
6. **Given** OTP 입력 시간(5분) 초과 시 **Then** "인증 시간이 초과되었습니다" 안내 + 재발송 버튼이 표시된다

### User Story 3 — 환자가 본인 상담 내역 조회 (Priority: P1)

로그인한 환자가 모든 연결된 치과의 상담 내역을 하나의 화면에서 시간 순으로 조회할 수 있다. 각 상담에는 내용, 첨부 사진, 처방 메모가 포함된다.

**Why this priority**: 환자가 서비스를 사용하는 핵심 목적.

**Independent Test**: 환자 로그인 → 상담 내역 목록 표시 → 항목 클릭 → 상세(내용·사진·처방메모) 조회 가능

**Acceptance Scenarios**:

1. **Given** 로그인한 환자가 **When** 상담 내역 페이지 접속, **Then** 연결된 모든 치과의 상담 이력이 최신순으로 표시된다
2. **Given** 상담 목록에서 특정 항목을 클릭하면 **Then** 상담 내용, 첨부 사진, 처방 메모 전체를 볼 수 있다
3. **Given** 연결된 치과가 2곳 이상인 환자가 **When** 상담 목록 조회, **Then** 기관명이 각 상담 항목 옆에 표시된다
4. **Given** 아직 상담 내역이 없거나 연결된 치과가 없는 경우, **Then** "아직 연결된 상담 내역이 없습니다" 안내가 표시된다

---

### User Story 4 — 재방문 환자 로그인 (Priority: P2)

이미 케어로그에 가입한 환자가 다시 서비스에 접속할 때, 초대 링크 없이 주민번호 + 전화번호 OTP로 로그인할 수 있다. 주민번호가 영구 계정 식별자 역할을 하므로 전화번호가 변경되어도 동일 계정에 접근할 수 있다.

**Why this priority**: 지속적 서비스 이용을 위해 필요하나 US1~3 이후 추가 가능.

**Independent Test**: 환자 포털 로그인 페이지 → 주민번호 + 전화번호 입력 → OTP 인증 → 상담 내역 화면 진입

**Acceptance Scenarios**:

1. **Given** 가입된 환자가 **When** 환자 포털에서 주민번호 + 전화번호 입력 + OTP 인증, **Then** 이전에 연결된 모든 치과 상담 내역이 표시된다
2. **Given** 전화번호가 바뀐 환자가 **When** 기존 주민번호 + 새 전화번호로 로그인, **Then** 새 번호로 OTP를 받아 동일 계정에 접근할 수 있다
3. **Given** 미가입 주민번호로 로그인 시도 시, **Then** "해당 정보로 가입된 계정이 없습니다. 치과에서 받은 링크로 가입해 주세요" 안내가 표시된다

---

### Edge Cases

- 같은 전화번호로 여러 치과에서 초대를 받았을 때, 동일 계정에 모두 연결되어야 한다
- 환자 레코드에 전화번호가 없는 경우 문자 전송 버튼이 비활성화된다
- 초대 링크는 발송 후 72시간 내에만 유효하다
- OTP는 5분 내 입력해야 하며 최대 3회 오류 시 10분 잠금
- 직원이 실수로 잘못된 번호로 발송한 경우 → 재발송 가능
- 환자가 계정을 탈퇴한 경우 연결된 치과 데이터 접근 불가 (기관 데이터는 보존)

## Requirements *(mandatory)*

### Functional Requirements

**직원 측 (SMS 발송)**
- **FR-001**: 직원은 환자 상세 페이지에서 환자 포털 초대 문자를 발송할 수 있어야 한다
- **FR-002**: 문자 발송 전 개인정보 제공 동의 확인 단계가 반드시 포함되어야 한다
- **FR-003**: 발송된 SMS에는 케어로그 가입 링크와 기관명이 포함되어야 한다
- **FR-004**: 초대 링크는 72시간 후 자동 만료되어야 한다
- **FR-005**: 동일 환자에게 재발송 시 이전 링크는 무효화되어야 한다

**환자 측 (가입 및 인증)**
- **FR-006**: 환자는 주민등록번호와 전화번호를 입력해 본인을 확인한 후 OTP 인증으로 로그인/가입할 수 있어야 한다
- **FR-007**: 주민등록번호 해시가 환자 계정의 영구 식별자가 되어야 한다 — 전화번호가 변경되어도 동일 계정에 접근할 수 있어야 한다
- **FR-008**: 가입 시 입력한 주민번호 해시가 초대장의 환자 기록과 불일치하면 OTP를 발송하지 않아야 한다
- **FR-009**: 동일 주민번호 해시로 이미 계정이 있으면 새 계정을 만들지 않고 기존 계정에 새 치과를 추가해야 한다
- **FR-010**: OTP는 발송 후 5분 이내에만 유효해야 한다
- **FR-011**: OTP 오류 3회 초과 시 10분 잠금이 적용되어야 한다
- **FR-012**: 주민등록번호가 등록되지 않은 환자에게는 초대 문자 발송 버튼이 비활성화되어야 한다

**환자 측 (상담 조회)**
- **FR-011**: 환자는 본인과 연결된 모든 치과의 상담 내역을 조회할 수 있어야 한다
- **FR-012**: 각 상담 항목은 기관명, 날짜, 상담 내용, 첨부 사진, 처방 메모를 포함해야 한다
- **FR-013**: 환자는 자신의 데이터만 조회할 수 있으며 다른 환자 데이터에 접근할 수 없어야 한다
- **FR-014**: 환자는 상담 데이터를 수정하거나 삭제할 수 없다 (읽기 전용)

### Key Entities

- **PatientAccount**: 주민번호 해시 기반 환자 계정. 여러 기관의 환자 레코드와 연결 가능. `(주민번호해시, 가입일, 최근 로그인)` — 전화번호는 저장하지 않음 (OTP 전달 수단일 뿐)
- **PatientInvitation**: 직원이 발송한 초대 기록. `(환자ID, 기관ID, 발송 전화번호, 동의 여부, 토큰, 만료 시각, 수락 여부)`
- **PatientAccountLink**: PatientAccount와 기관 내 환자 레코드의 연결. `(PatientAccount, patient 레코드, 기관ID, 연결 시각)`

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 환자가 SMS 수신 후 5분 이내에 계정 가입 및 상담 내역 조회를 완료할 수 있다
- **SC-002**: 직원의 문자 발송 동작이 현재 상담 흐름에서 2단계(전화번호 입력 + 동의 체크) 이내로 완료된다
- **SC-003**: 이미 계정이 있는 환자가 새 치과 초대 링크를 통해 추가 연결 시 별도 재가입 없이 처리된다
- **SC-004**: 환자 계정으로 로그인한 사용자가 다른 기관 또는 다른 환자의 데이터에 접근할 수 없음이 검증된다

## Assumptions

- SMS 발송에는 외부 문자 발송 서비스(국내 기준: Solapi, CoolSMS 등)가 필요하며 API 키 설정이 사전에 완료된다
- 환자 전화번호는 한국 기준 010-XXXX-XXXX 형식을 기본으로 한다
- 환자 포털은 모바일 웹 브라우저에서 우선 동작하도록 최적화한다 (별도 앱 불필요)
- 환자 계정은 직원/원장 계정과 완전히 분리된 별도 계정 유형이다
- 환자 포털은 읽기 전용이며 v1에서는 예약, 메시지 등의 기능을 포함하지 않는다
- 개인정보보호법 준수를 위해 동의 없이는 환자 데이터를 환자 본인에게도 전송하지 않는다
- 기존 `patient` 테이블의 데이터는 그대로 유지되며, 환자 계정과의 연결은 별도 링크 테이블로 관리한다
- **주민번호가 등록된 환자만** 포털 초대가 가능하다. `resident_no_hash`가 없는 환자는 초대 버튼 비활성화
- 주민번호 해시(`resident_no_hash`)는 기존 `rrn-hash.ts`의 SHA-256 로직을 재사용한다
- 환자 계정에 전화번호를 저장하지 않는 이유: 전화번호는 변경 가능하므로 영구 식별자로 부적합. OTP 발송 시마다 입력받는 방식으로 처리
