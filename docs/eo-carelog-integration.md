# EO ↔ Carelog 층위 · 통합 계약 (논의 메모)

> 상태: **방향 합의 단계** (실비 카드224 회신 반영 · 일부 미결) · 작성: 다온(Carelog) · 갱신: 2026-06-07 세션15
> 관련: 제안 카드 25 (EO 전달) → **회신: 실비 카드224** / EO 관문계약: 헤임달 카드219·spec-048 (Carelog 레포 미보유 → EO측 확인 필요)

Carelog는 EO의 광범위한 병의원 솔루션과 **환자**를 잇는 다리(bridge)다.
두 제품은 별도 Vercel·Supabase로 **격리 운영**되므로, 상호작용은 "경계(계약)"로만
이뤄져야 한다. 이 문서는 그 경계를 정하기 위한 제안과 미결 질문을 담는다.

## 1. 현재 상태 (코드 근거)

Carelog는 자체적으로 다음을 보유한다:
- `institutions`, `institution_members`(직원), `patient`, `chairs`, `consultation`
- 환자 평생계정: `patient_accounts`(rrn_hash 유니크 키), `patient_account_links`, 세션·OTP·푸시

→ EO도 병의원/직원 마스터를 가진다면 **같은 마스터가 두 곳 = SSOT 위반** 위험.

## 2. 제안 — SSOT 분할 + 브리지 계약

```
        ┌──────────────┐   직원/클리닉 마스터(EO→CL)   ┌──────────────┐
        │     EO       │ ───────────────────────────▶ │   Carelog    │
        │ 병의원 운영   │                               │ 환자 engagement│
        │ SSOT         │ ◀─────────────────────────── │ SSOT          │
        │(클리닉·직원·  │   환자 신호(CL→EO)            │(환자계정·동의· │
        │ 분석)        │   피드백·동의·engagement       │ 기록열람·SMS) │
        └──────────────┘                               └──────────────┘
              └── 공유 "표준"(DB는 격리): 암호화 lib/KMS · RRN 해시 · 동의 스키마 ──┘
```

- **EO = 병의원 운영 SSOT** — 클리닉/직원 명단/B2B 분석(광범위 솔루션)
- **Carelog = 환자 engagement SSOT** — 환자 평생계정·동의·진료기록 열람·SMS/포털·피드백
- **브리지(격리 유지, 계약으로만 연결)**
  - EO→Carelog: 직원/클리닉 **마스터 제공**(피드백 직원선택이 항상 실제 명단)
  - Carelog→EO: 환자 **신호 이벤트**(피드백·동의·engagement)
- **보안**: EO 암호화 라이브러리/KMS·RRN 해시 표준을 Carelog가 **재사용**(재발명 금지).
  단, 데이터 저장소는 격리(PLAYBOOK §5 격리 우선).

> 단기 현실: 직원 명단이 Carelog에도 있으므로 피드백 기능은 **Carelog 내 구현 +
> EO로 이벤트 1건 송신**으로 시작 가능. "마스터 EO 이관"은 더 큰 결정이라 분리.

## 3. 파일럿 — 진료 후 피드백 (첫 CL→EO 연동)

- 진입: 기존 상담 SMS/포털 진입에 얹기(별도 발송 X) — "오늘 진료 어떠셨어요?"
- 흐름: 칭찬/아쉬움 토글 → (칭찬 시) 직원 다중선택 → 선택적 한 줄 → 1화면 종료
- 프라이버시(정책 미결): 칭찬=직원/클리닉 귀속 노출 / 불편=클리닉엔 비식별 집계,
  EO엔 에스컬레이션 식별 여부 = **EO 합의 필요**
- 넛징: 피드백 직후 "내 진료기록 받아보기"로 Carelog 계정 유도(단계적 가입과 연결)
- 피로 방지: 매 방문 X, 간격 가드

### 4.1 확정으로부터 도출되는 설계 제약
- **마스터 EO 이관**: Carelog의 `institutions`/`institution_members`는 장기적으로 EO 마스터의 **읽기 투영(projection)** 으로 전환. 단기는 §2 단서대로 CL 내 보유 + 이벤트 송신 유지, 이관은 별도 마이그레이션 결정.
- **환자 SSOT=CL**: `patient_accounts`(rrn_hash) 등 환자 계정 계열은 Carelog 소유 유지 — 변경 없음.
- **관문 일원화**: EO↔CL 모든 호출은 **헤임달 게이트웨이** 단일 진입점 경유. Carelog는 테오/개별 EO 서비스에 직접 붙지 않음.
- **인증 재사용**: 신규 토큰 발명 금지. EO `sso-token` + `workspace_carelog_links` 매핑을 **재사용**. (토큰 발급/검증 흐름 = 헤임달 계약서 확인 후 확정)

## 5. 브리지 계약 초안 (지시3 · 1차)

> Carelog 소유 영역(환자 이벤트 스키마)은 여기서 정의 가능. EO 소유 영역(마스터 페이로드·헤임달 인증)은 **TBD(헤임달 계약 확인)** 로 표기.

### 5.1 EO → CL : 직원·클리닉 마스터 (읽기 방향)
- **목적**: 피드백 직원선택이 항상 실제 명단과 일치
- **전송**: 헤임달 관문 경유 (방식 = pull API vs push sync, **TBD**)
- **인증**: `sso-token` / `workspace_carelog_links` 재사용 (**TBD**: 서비스-서비스 토큰 스코프)
- **페이로드(EO 정의 영역, TBD)**: `clinic_id`, `staff_id`, 표시명, 역할, 활성여부, `valid_from/to`

### 5.2 CL → EO : 환자 신호 이벤트 (Carelog 정의 영역, 1차 제안)
- **전송**: 헤임달 관문 경유 단방향 이벤트
- **이벤트 타입(초안)**:
  - `feedback.submitted` — 진료 후 피드백 (§3 파일럿)
  - `consent.updated` — 환자 동의 상태 변경
  - `engagement.signal` — 포털/SMS 열람 등 (후순위)
- **공통 봉투(초안)**:
  ```jsonc
  {
    "event": "feedback.submitted",
    "version": "1",
    "occurred_at": "<ISO8601>",
    "source": "carelog",
    "clinic_id": "<EO 마스터 식별자>",
    "idempotency_key": "<uuid>",     // 재전송 안전
    "payload": { /* 타입별 */ }
  }
  ```
- **`feedback.submitted` payload(초안)**:
  ```jsonc
  {
    "visit_ref": "<상담/방문 참조>",
    "sentiment": "praise | concern",
    "staff_ids": ["..."],            // 칭찬 시 직원식별 O (Q4 확정)
    "comment": "<선택, 한 줄>",
    "patient_identified": false       // 불편 케이스 비식별 경계 = Q4 세부 확정 후
  }
  ```
- **인증/서명·재시도·DLQ**: 헤임달 계약서 확인 후 확정 (**TBD**)
