## 0. 문제 / 목표

기존 체어 즉시기록(006)은 **직원 측** 마찰(환자 선택 → 화면 이동 → 녹음)을 없앴다.
남은 마찰은 **진료 현장의 사회적·심리적 부담**이다:

- 직원: "지금 녹음해도 될까요?"를 매번 말로 꺼내기 어색함 → 기록 누락
- 환자: 갑자기 녹음한다고 하면 경계 → 신뢰 저하

목표: **진료실 모니터에 띄우는 환자 설명화면 + 원탭 빠른녹음 + 정형화된 의료진 멘트**로,
녹음·기록을 자연스럽고 투명한 진료 루틴으로 만든다.

---

## 1. 산출물 ① — 환자 설명화면 + '빠른 녹음' 버튼

### 1.1 무엇 / 어디에
- 환자가 체어에서 바라보는 **진료실 모니터**에 띄우는 읽기 전용 설명화면.
- 직원이 진료 시작 시 한 번 보여주고, "빠른 녹음" 한 번으로 기록 시작.

### 1.2 화면 구성 (잠정 카피 — 달리 브리프로 대체)
```
┌──────────────────────────────────────────┐
│  오늘 진료, 기록으로 남겨드릴게요          │   ← 헤드라인
│                                            │
│  · 상담 내용을 음성으로 안전하게 기록해요   │
│  · 진료 후 '내 진료기록'으로 다시 보실 수    │   ← 가치 3줄
│    있어요                                   │
│  · 기록은 병원과 환자분만 볼 수 있어요       │
│                                            │
│        [ 🎙  빠른 녹음 시작 ]               │   ← 단일 CTA
│        오늘은 기록 없이 진료할게요 (작게)    │   ← 거절 경로
└──────────────────────────────────────────┘
```
- **단일 CTA 원칙**: 버튼 하나(빠른 녹음). 거절은 작은 텍스트 링크로 항상 존재(강요 X).
- 녹음 중 전환 시 화면은 **"기록 중 ●"** 최소 표시로 바뀌어 환자가 진행 상태를 인지(투명성).

### 1.3 동작
1. 모니터: 기본=설명화면 → 직원이 "빠른 녹음" 탭 → 모니터 "기록 중 ●" 표시
2. 실제 녹음·AI 변환·임시저장은 **006 체어 즉시기록 파이프라인 재사용** (신규 녹음엔진 만들지 않음)
3. 종료 시 모니터는 "기록 완료, 진료 후 받아보실 수 있어요"로 마무리 + (선택)포털 가입 넛징

---

## 2. 산출물 ② — 의료진 멘트 (스크립트)

> 잠정 초안. 달리 브리프 멘트로 대체. 3초 안에 끝나는 자연스러운 한두 마디 지향.

| 상황 | 멘트(초안) |
|---|---|
| 녹음 권유 | "오늘 상담 내용 기록으로 남겨드릴게요. 화면 한번 보실게요." |
| 동의 확인 | "괜찮으시면 시작할게요." (환자 끄덕임/구두 OK → 직원이 시작 탭) |
| 거절 시 | "네, 기록 없이 진행할게요." (부담 0, 즉시 설명화면 닫기) |
| 종료 | "기록 끝났어요. 진료 끝나고 휴대폰으로도 보실 수 있어요." |

- 원칙: **권유→짧은 동의 확인→시작**. 설득·강요 금지. 거절은 1초 안에 받아들이는 멘트.

## 3. 다온 결정사항 (카드229 위임: 모니터 표시 · 동의 저장 · 기기 플로우)

### 3.1 모니터 표시방식 — **2기기 + Realtime 동기화**
- 환자 모니터에 **로그인 불요 읽기 전용 라우트** `(/present/[chairId])` 를 띄운다.
  기관·체어 단위 **표시 토큰**으로 접근(직원 세션·환자 개인정보 노출 X).
- 직원 기기(컨트롤)와 모니터는 **Supabase Realtime 채널**(`chair:{id}:present`)로 상태 동기화
  → 직원이 "빠른 녹음" 탭하면 모니터가 즉시 "기록 중"으로 전환.
- 모니터는 **표시 전용**(녹음 자체는 직원 기기 또는 006 파이프라인). 모니터에서 환자 데이터 접근 없음.

### 3.2 동의 저장 — **'녹음·기록 동의'를 진료 시점에 캡처 (개인정보 제공 동의와 분리)**
- 신규 개념: 기존 `patient_account_links.consent_given`(개인정보 제공)과 **다른** "녹음/기록 동의".
- 저장 위치: `consultation`(또는 체어 임시기록)에 컬럼 추가 제안 —
  - `recording_consent` boolean
  - `consent_method` text — 초기값 `'verbal_onscreen'`(구두+화면 표시)
  - `consent_at` timestamptz
  - (선택) `consent_by` — 동의 확인한 직원 계정
- **거절도 기록**: `recording_consent=false`로 남겨 감사·재권유 가드에 활용.
- 환자계정이 없어도(미가입) 진료 시점 동의가 성립 — 환자 SSOT=CL 구도와 정합(카드224).
- 감사: 동의/거절 이벤트를 `chair_audit_logs`에 이벤트 유형으로 추가.

### 3.3 기기 플로우 — **2기기 기본 · 1기기 폴백**
- **기본(2기기)**: 직원 기기=컨트롤(빠른녹음/멈춤), 체어 모니터=환자 설명·상태 표시.
- **폴백(1기기)**: 모니터 없는 체어 → 직원이 자기 기기를 환자에게 돌려 설명화면 보여주는
  "보여주기 모드". 동일 라우트의 컴팩트 레이아웃 재사용.
- 빠른녹음 버튼의 **실행 주체는 직원**(오조작·환자 개인정보 보호). 모니터는 미러링만.
