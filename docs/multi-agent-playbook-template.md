# 멀티 에이전트 협업 플레이북 (이식용 템플릿 · 원격 웹세션판)

> 출처: EO 프로젝트에서 검증된 다중 Claude Code 에이전트 협업 모델을 **다른 프로젝트(별도 git 레포)** 에 이식하기 위한 범용 지침.
> 이 판본은 **Claude Code 웹(원격 컨테이너) 실행 환경** 기준이다. (로컬 워크트리 방식은 생략)
> 사용법: 이 파일을 새 프로젝트 레포의 `docs/`에 두고 → `{{중괄호}}` placeholder를 프로젝트 값으로 치환 → §11 체크리스트 수행.
> 새 프로젝트의 셋업 담당 에이전트(또는 사람)에게 이 문서 전체를 그대로 전달하면 됨.

---

## 0. 한 장 요약 (TL;DR)

- **역할이 분리된 N개 에이전트**가 **각자 전용 git 브랜치**에서 동시에 일한다.
- 에이전트끼리 **직접 대화하지 않는다** — 사람이 중계하고, 정보는 **git 파일**(spec/tasks/status)로 공유된다.
- **공통 규칙은 `CLAUDE.md`(전 브랜치 동일)**, **에이전트 정체성은 `CLAUDE.local.md`(브랜치마다 다름, `.gitattributes`로 머지 보호)** 로 분리한다.
- 모든 커밋은 **에이전트 말머리**(`[이름]`)로 시작해 누가 한 일인지 추적한다.
- **문서 담당(기획)과 코드 담당(개발)의 파일 소유권을 엄격히 분리**해 충돌을 구조적으로 없앤다.

---

## 1. 핵심 철학 (왜 이렇게 하는가)

이식할 때 형식만 베끼지 말고 아래 원칙을 이해할 것. 프로젝트가 달라도 이 원칙은 그대로다.

1. **역할 분리 = 충돌 제거** — 각 에이전트의 *파일 소유 영역*이 겹치지 않으면 머지 충돌이 구조적으로 안 난다. (기획자는 `docs/`·`specs/`, 개발자는 `src/`)
2. **단일 진실원(single source)은 git** — 에이전트는 휘발성. 영속 정보·핸드오프·결정은 전부 git 파일에 남긴다. "말로만" 공유 금지.
3. **사람이 라우터** — 에이전트는 서로 못 부른다. 사람이 한 에이전트의 산출물을 다른 에이전트에게 전달한다. 그래서 핸드오프 문서가 명확해야 한다.
4. **정체성은 로컬, 규칙은 공통** — 정체성 파일을 브랜치별로 두되 머지 시 덮어쓰이지 않게 보호. 공통 규칙은 한 파일에서 전 브랜치로 전파.
5. **추적 가능성** — 커밋 말머리로 "누가/어떤 역할로" 했는지 한눈에. 배포 대시보드·로그에서 바로 식별.

## 2. 에이전트 구성 (역할 분리)

> 아래는 4역할 기본형. 프로젝트 규모에 맞게 가감하되 **"기획↔개발 분리"는 최소 유지** 권장.

| 역할 유형 | 책임 | 소유 파일 | 코드 수정 |
|-----------|------|----------|----------|
| **기획 PM** | 스펙·UX·로드맵·우선순위 | `specs/`, `roadmap.md`, `*status.md`, `vision.md` | ❌ (읽기만) |
| **전략 검토** | 기획 토론·리서치·대안·리스크 검토 | 기획 PM과 동일(동시편집 금지) | ❌ (읽기만) |
| **시니어 개발** | 구현·DB·배포·인프라 | `src/`, `package.json`, DB 마이그레이션, 설정 | ✅ (문서는 읽기만) |
| **주니어 개발** | 가벼운 기능·UI·버그 — 시니어와 병렬 | `src/`(시니어와 충돌 없는 파일) | ✅ (문서·배포 금지) |

**커스터마이즈 포인트:**
- 1인 프로젝트면 "기획 PM + 시니어 개발" 2역할로 시작 가능.
- 각 에이전트에 **이름 + 색상 이모지**를 부여(예: 🟣🟢🔵🔹)하면 사람이 식별하기 쉽다.

## 3. Git 브랜치 전략

### 기본 구조

```
main  ─ 프로덕션 (배포 대상)
dev   ─ 통합/개발 (항상 여기서 합류)

기획PM   : {{plan}}/docs   → main → dev 동기화
전략검토 : {{review}}/docs → main → dev 동기화
시니어개발: dev            → main 배포
주니어개발: {{junior}}/dev → dev → main
```

원칙:
- **각 에이전트는 자기 전용 브랜치에서만 작업**한다. 남의 브랜치에 직접 push 금지.
- 문서 브랜치는 `main` 머지 후 `dev`에 동기화(또는 프로젝트 규약에 맞게).
- 개발 브랜치는 `dev`에서 통합 후 `main` 배포.
- **배포(main 머지)는 "개발·배포 담당" 1명의 전속 권한.** 기획자는 main에 직접 머지하지 않는다.

### 실행 환경 — 원격 웹세션 (Claude Code on the web)

- 세션마다 **격리된 임시 컨테이너** + 에이전트별 **작업 브랜치 지정**으로 동작한다. 로컬 워크트리·폴더 분리 불필요.
- 컨테이너는 **휘발성** — 작업이 끝나면 반드시 자기 브랜치에 `commit` + `push`. 안 하면 사라진다.
- 레포는 세션 시작 시 **새로 클론**된다. 따라서 `.gitattributes`·`CLAUDE.md`·`CLAUDE.local.md` 등 **설정·정체성은 반드시 레포에 커밋**돼 있어야 다음 세션에서도 유지된다.
- 한 에이전트가 한 세션에서 자기 브랜치만 다룬다 → 동시 편집 충돌은 "같은 파일을 두 에이전트가 같은 시기에" 건드릴 때만 발생(§5-1로 회피).

### 커밋 컨벤션 (말머리 필수)

```
[기획PM] chore: 스펙 문서 작성
[전략검토] chore: 전략 검토 메모
[시니어] feat: 기능 구현
[시니어] fix: 버그 수정
[주니어] feat: 가벼운 기능
```

→ 배포 대시보드/로그에서 작성자 즉시 식별. 타입(feat/fix/chore/refactor)은 conventional commits 따름.

## 4. CLAUDE.md vs CLAUDE.local.md 분리 (핵심 메커니즘)

| 파일 | git 추적 | 브랜치별 차이 | 내용 |
|------|----------|--------------|------|
| `CLAUDE.md` | ✅ 전 브랜치 **동일** | 없음 | 프로젝트 공통: 워크플로·역할 정의·브랜치 표·커밋 컨벤션·협업 규칙·외부 시스템 |
| `CLAUDE.local.md` | ✅ 브랜치마다 **다름** | 있음 | **에이전트 정체성** + 그 에이전트 전용 세션 마무리 루틴 |
| `.gitattributes` | ✅ | — | `CLAUDE.local.md merge=ours` — 머지 시 대상 브랜치 것이 항상 이김 |

### 머지 충돌이 안 나는 이유

```
기획PM이 자기 브랜치 → main 머지
  ↓ .gitattributes의 merge=ours 발동
main의 CLAUDE.local.md(개발담당 정체성) 자동 유지 → 충돌 0, 수동개입 0
```

설정(레포에 1회 커밋 + 머지 수행 환경마다 driver 등록):
```bash
# 1) 레포에 커밋 (한 번): .gitattributes 에 한 줄
echo 'CLAUDE.local.md merge=ours' >> .gitattributes

# 2) merge driver 등록 (머지를 실제 수행하는 컨테이너/환경마다)
#    원격 웹세션은 컨테이너가 휘발하므로, 머지를 돌리기 전에 한 번 실행.
git config merge.ours.driver true
```

> ⚠️ `merge=ours` 속성(파일별)과 `-s ours`(전체 전략)는 다름. 위는 **파일별** 보호라 나머지 파일은 정상 머지된다.
> ⚠️ 원격 컨테이너는 매 세션 새로 뜨므로 global config가 초기화될 수 있다. **머지 작업(보통 배포 담당)을 시작하기 직전 `git config merge.ours.driver true`를 한 번 실행**하면 안전하다.

## 5. 협업 규칙 (CLAUDE.md에 박을 것)

EO에서 검증된 규칙. 새 프로젝트 CLAUDE.md에 그대로 넣고 시작:

```
1. 동일 파일 동시 수정 금지 — 작업 전 어떤 파일 건드는지 공유(특히 개발자끼리).
2. 핸드오프 — 기획자가 specs/NNN/tasks.md 완성 → 개발자가 확인 후 구현 착수.
3. 작업 시작 전 반드시 git pull origin <내 브랜치> (원격 세션은 클론 직후 최신이지만, 장기 세션 중엔 fetch).
4. 기획자는 기능 완료 시 projectstatus.md(사실 기록) 업데이트.
5. 개발자는 구현 완료 시 tasks.md 해당 항목 완료 표시.
6. DB 마이그레이션·인프라 변경은 개발·배포 담당 확인 후 진행.
7. CLAUDE.local.md는 각 브랜치에 커밋 — .gitattributes merge=ours로 보호.
8. 새 spec/문서 채번 전 전 브랜치 확인 필수:
   - main만 보면 안 됨(다른 에이전트 작업이 아직 main에 없을 수 있음).
   - git fetch origin → 전 브랜치 통틀어 최고 번호 +1.
   - 충돌 시 늦게 만든 쪽이 양보.
```

### 규칙 8 실행 명령 (채번 전 1줄)

```bash
git fetch origin --quiet && \
for b in $(git branch -r | grep -v HEAD); do \
  git ls-tree -d --name-only "$b" -- specs/ 2>/dev/null; \
done | grep -oE 'specs/[0-9]{3}' | grep -oE '[0-9]{3}' | sort -un | tail -1
# 출력값 + 1 이 다음 번호
```

## 6. 파일 소유권 경계 (충돌 방지의 핵심)

```
기획PM / 전략검토  ─ specs/, roadmap.md, *status.md, vision.md  (src/ 읽기만)
시니어 / 주니어개발 ─ src/, 설정, DB 마이그레이션              (specs/ 읽기만)
```

- 기획자가 코드를 못 만지고, 개발자가 기획 문서를 안 고치면 **두 그룹의 머지가 영원히 안 부딪힌다.**
- 같은 그룹 안(개발자 2명)에서는 §5-1(작업 파일 사전 공유)로 조율.

## 7. 에이전트 간 정보 전달

직접 소통 없음. 두 경로만:

**(A) 사람이 중계** — "기획자가 만든 specs/NNN/tasks.md 보고 구현해줘"처럼 사람이 산출물을 다른 에이전트에 전달.

**(B) git 파일** — 핸드오프 큐/상태 문서로 비동기 공유:
```
specs/NNN/spec.md      기능 스펙 (기획자 작성)
specs/NNN/tasks.md     개발 작업지시 (기획자 작성 → 개발자가 완료 체크)
projectstatus.md       사실 기록 (완료된 것)
roadmap.md             우선순위·방향
specs/000-backlog/*    핸드오프 큐·협업 메모
```

> 핸드오프가 명확할수록 사람의 중계 부담이 준다. "다음에 누가 뭘 해야 하는지"를 항상 문서에 남길 것.

## 8. 세션 루틴

### 시작 (모든 에이전트 공통)
```bash
git pull origin <내 브랜치>   # 원격 세션은 클론 직후면 생략 가능
# (프로젝트별 시작 점검 스크립트가 있으면 실행)
```

### 마무리 — 기획/전략(문서) 담당
```bash
git add specs/ roadmap.md projectstatus.md
git commit -m "[{{이름}}] chore: 세션 마무리 문서 업데이트"
git push origin <내 브랜치>
# main 머지는 하지 않음 — 배포 담당 영역
```

### 마무리 — 개발·배포 담당 (배포 권한 보유자만)
```bash
git config merge.ours.driver true   # 원격 컨테이너면 머지 전 1회
git push origin dev
git checkout main && git merge dev --no-edit && git push origin main && git checkout dev
# 새 기능이 있으면: 버전업 + CHANGELOG + 릴리즈 draft (프로젝트 규약)
```

> 각 에이전트의 **구체 루틴은 자기 `CLAUDE.local.md`에** 적어둔다(아래 템플릿). 사람이 "마무리해줘"라고 하면 자기 루틴만 실행.

## 11. 부록 C — 신규 프로젝트 적용 체크리스트 (원격 웹세션)

```
[ ] 1. 레포에 main, dev 브랜치 준비
[ ] 2. .gitattributes 에 'CLAUDE.local.md merge=ours' 추가 → 레포에 커밋
[ ] 3. CLAUDE.md 작성 (부록 A 템플릿 → placeholder 치환, 협업규칙 1~8 포함) → main 커밋
[ ] 4. 에이전트별 브랜치 생성 (기획/전략/개발…)
[ ] 5. 각 브랜치에 CLAUDE.local.md 작성·커밋 (부록 B 템플릿)
[ ] 6. (원격 웹세션) 에이전트별 작업 브랜치를 세션에 지정 — 워크트리 불필요
[ ] 7. specs/ 디렉토리 + 000-backlog(핸드오프 큐) 골격 생성
[ ] 8. projectstatus.md, roadmap.md 초기 파일 생성
[ ] 9. 첫 핸드오프 테스트: 기획자가 specs/001 작성 → push → 개발자가 다른 세션에서 pull 후 확인
[ ] 10. 배포 담당: 머지 전 'git config merge.ours.driver true' 1회 실행 습관화
```

## 12. 흔한 함정

- **채번 충돌**: 규칙 8을 안 지키면 두 에이전트가 같은 번호 spec을 만든다. *반드시 fetch 후 전 브랜치 확인.*
- **main 직접 머지**: 기획자가 배포까지 해버리면 책임 경계가 무너진다. 배포는 개발·배포 담당 전속.
- **말로만 핸드오프**: git 파일에 안 남기면 다음 세션/다른 에이전트가 모른다. 결정·핸드오프는 항상 문서화.
- **CLAUDE.local.md 미커밋**: 브랜치에 정체성 파일이 없으면 에이전트가 자기가 누군지 모른다. 브랜치 생성 직후 커밋.
- **merge driver 누락**: 원격 컨테이너는 매 세션 초기화 → 머지 전 `git config merge.ours.driver true` 미실행 시 CLAUDE.local.md가 충돌난다. 배포 담당 루틴에 포함.
- **push 누락**: 원격 컨테이너는 휘발 → 세션 끝에 push 안 하면 작업이 사라진다.
