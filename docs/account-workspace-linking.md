# 계정·워크스페이스 연동 규칙 (EO ↔ Carelog)

> 목적: "Carelog로 먼저 들어온 사용자가 EO와 어떻게 연결되는가"와 그때 필요한 **수동 운영**을
> 한곳에 못박아 둔다. (세션 대화에만 있으면 휘발되므로 SSOT로 보존)
> 작성: 다온 · 2026-06-20 · 코드 근거: `app/api/auth/sso/route.ts`, `app/actions/auth.ts`,
> `lib/auth/institution.ts`, EO 게이트웨이(`lib/eo/gateway.ts`)·`workspace_carelog_links`(EO 소유)

---

## 0. 한 줄 요약

- **계정 식별 키 = 이메일** (Supabase Auth: 이메일 1개 = 사용자 1개)
- **EO↔CL 연동 = '기관(워크스페이스)' 단위** — EO가 자기 클리닉을 **특정 Carelog `institution_id`** 에 링크
- **사용자 합류 ≠ 워크스페이스 연동 이전** — 멤버십만 붙고, 본인이 만든 기관은 EO에 딸려가지 않음
- 그래서 **중복/빈 기관 정리·재링크·데이터 이관은 수동**(자동화 없음)

---

## 1. 직원 진입 경로 3가지

| 경로 | 결과 | 비고 |
|---|---|---|
| **EO SSO** (`/api/auth/sso`, "케어로그 열기") | JWT의 `institution_id`(= EO가 링크해 둔 기관)에 **멤버로 합류** + `eo_employee_id`·표시명 기록 | EO가 직원으로 알고 있어야 함 |
| **이메일+비밀번호** (`/signup`→`/login`) | **본인 기관을 새로 생성**하고 owner | 공개(EO 검증 없음). `signUp`(`app/actions/auth.ts`) |
| **Google OAuth** (`/login` 버튼) | 멤버 없으면 온보딩 → 새 기관 생성 | 공개 |

> 환자 포털(`/portal/*`)은 SMS/OTP 기반의 **완전히 별개** 경로 — 직원 로그인과 무관.

---

## 2. 연동의 핵심: '기관 단위' 링크

EO↔Carelog 연동은 **EO 클리닉 ↔ 특정 Carelog `institution_id`** 매핑이다.
(EO 소유 테이블 `workspace_carelog_links`, 게이트웨이는 `GET /api/gateway/carelog/master?institution_id=<CL기관ID>` 로 호출, SSO JWT도 그 `institution_id`를 실어 보냄.)

```
[사용자] foo@clinic.com  (Auth 계정 1개 — 이메일이 키)
   ├── 기관 X  ← 본인 셀프가입 때 만든 워크스페이스   (EO 연동 ❌, 독립)
   └── 기관 Y  ← EO가 자기 클리닉에 링크해 둔 기관      (EO 연동 ✅)
                  └ SSO 진입 시 여기에 '멤버'로만 추가
```

→ **연동성(직원 마스터 동기화·작성자 귀속 등)은 기관 Y에만** 작동한다. 사용자가 합류해도
**기관 X가 EO로 끌려 들어가지 않는다.** SSO 라우트는 JWT의 `institution_id`에만 멤버십을
넣고, 사용자의 다른 기관은 건드리지 않는다.

---

## 3. 시나리오별 결과

| 상황 | 결과 |
|---|---|
| 같은 이메일 + EO SSO 1회 진입 | 기존 계정에 **EO 기관(Y) 멤버십 자동 추가** → 기관 2개, 헤더 드롭다운 전환 |
| 이메일이 다름 | **연결 안 됨** — EO가 그 이메일을 직원으로 알고 있어야 함 |
| 셀프로 "내 계정을 EO에 잇기"? | **불가** — EO 쪽 직원 등록 + SSO/마스터 동기화로만 연결됨 |
| 기관 X에 미리 만든 환자·상담 | EO로 **자동 이관 안 됨** (멀티테넌트 RLS 격리 — X 데이터는 X에만) |

---

## 4. 수동 운영이 필요한 경우 (규칙)

자동화가 없으므로 아래는 **사람이 판단·실행**한다.

### 4.1 중복/빈 기관 정리
- 증상: EO 직원이 셀프 가입까지 해서 **빈(또는 반쪽) 기관 X**가 남음.
- 조치: 사용자를 EO 연동 기관(Y)에서 쓰게 하고, **X는 비활성/삭제**. (현재 셀프 정리 UI 없음 → DB/관리자 수동)

### 4.2 "내 기존 기관(X)을 EO 연동 기관으로 쓰고 싶다"
- 조치: **EO 쪽에서 링크 대상을 X의 `institution_id`로 재지정**(`workspace_carelog_links` 갱신).
- 주의: 이미 다른 기관 Y가 링크돼 있으면 **중복/충돌 정리** 동반. EO 담당(헤임달/테오)과 협의.

### 4.3 데이터 이관 (X → Y)
- X에 실제 환자·상담이 쌓였는데 Y로 합쳐야 하면, `institution_id`를 옮기는 **수동 마이그레이션** 필요.
  (환자 중복·RRN 해시 충돌·작성자 귀속 재계산 고려. 일회성 SQL로 처리, 사전 백업.)

---

## 5. 권장 운영 (사고 예방)

- **EO 소속 직원은 처음부터 "케어로그 열기"(SSO)로만 진입** → 곧장 EO 연동 기관(Y)에 안착, 기관 X가 아예 안 생김.
- 파일럿/초기엔 **공개 가입을 막고 EO·초대 기반으로 잠그는** 것도 방법(아래 §7).

---

## 6. 확인 필요(엣지 케이스)

- **Google 가입 + 동일 이메일 EO SSO**: Supabase "identity linking" 설정에 따라 매직링크(SSO)가
  같은 계정에 붙는지 **검증 필요**. (이메일/비번 가입은 `generateLink`가 기존 계정을 그대로 찾아 문제없음.)
  실제 케이스 발생 시 1회 검증할 것.

---

## 7. 후속 옵션 (코드로 '규칙'을 강제하려면)

지금은 이 문서가 규칙의 SSOT다. 더 단단히 하려면:
1. **공개 가입 차단** — `/signup`·Google 비활성 또는 EO/초대 화이트리스트만 허용(애초에 기관 X 미생성).
2. **빈 중복 기관 정리 도구** — owner가 자기 빈 워크스페이스를 닫거나, 관리자가 병합.
3. **온보딩 가드** — EO 이메일 도메인/마스터에 있는 사용자가 셀프 가입을 시도하면 "EO에서 케어로그 열기로 들어오세요" 안내·차단.

> 위는 미구현. 필요 시 별도 spec으로 착수.
