# PLAYBOOK — 조직 공통 운영 플레이북

> **버전**: 2026-05-31 (세션 55) · 원본: EO 레포
> **적용 대상**: 전 제품(EO · Carelog · Iris …) 공통. 제품 무관한 **일하는 방식·협업·에이전트·철학**만 담는다.
> **제품별 특화**(브랜치·배포 URL·외부 시스템·디자인 토큰·도메인 지식)는 각 제품 레포의 `CLAUDE.md`에 둔다.
> **동기화**: 이 파일이 바뀌면 전 제품 레포에 같은 내용을 반영한다 (§7 동기화 프로토콜).

## 0. 이 문서의 위치

- 각 제품 레포 루트에 **같은 `PLAYBOOK.md` 한 부씩** 둔다 (에이전트는 자기 레포 파일만 읽으므로).
- 각 제품 `CLAUDE.md` 최상단에서 이 파일을 먼저 읽도록 가리킨다:
  > "조직 공통 운영 규칙은 `PLAYBOOK.md`를 먼저 읽을 것. 이 `CLAUDE.md`는 본 제품 특화 사항만 다룬다."

## 1. 커뮤니케이션 규칙 (전 에이전트 공통 — 필수)

1. **호칭·존댓말** — 사용자는 **송정현 대표**다. 이름(송정현)은 인지하되, 부를 때는 항상 **"대표님" 또는 "송대표님"** 으로 부르고 **존댓말**을 쓴다. (반말·이름 호칭 금지)
2. **작업지시는 "복붙 1~2줄"로 정리해 드린다** — 대표님이 다른 에이전트에게 작업을 넘겨야 하는 상황이면, 에이전트가 **모든 맥락(결정·수용기준·순서)을 spec/tasks/핸드오프 문서에 미리 정리**해두고, 대표님께는 **그대로 복사해 전달할 수 있는 핸드오프 카드**를 명확히 따로 제시한다. 대표님이 긴 맥락을 재설명할 필요가 없어야 한다.

## 3. 커밋 컨벤션 (전 제품 공통)

모든 커밋 메시지 맨 앞에 **에이전트 말머리**를 붙인다. 배포 대시보드에서 누가 작업했는지 바로 파악.

```

[테오] fix: 버그 수정
[테오] refactor: 리팩토링
[렉스] feat: 가벼운 기능

[실비] chore: 스펙 문서
[레아] chore: 전략 검토 문서
[다온] feat/fix/chore: …
```

| Theo | `[테오]` | feat, fix, refactor, chore |

| Sylvie | `[실비]` | chore |

## 4. 협업 규칙

1. 동일 파일 동시 수정 금지 — 작업 전 어떤 파일 건드는지 확인.
2. 핸드오프: 기획자(Sylvie/Leah)가 tasks 완성 → 개발자(Theo/Rex)가 확인 후 구현 착수.
3. 작업 시작 전 반드시 `git pull origin <내 브랜치>` 실행.
4. 기획자는 기능 완료 시 상태 문서(projectstatus 등) 업데이트.
5. 개발자는 구현 완료 시 tasks 해당 항목에 완료 표시.
6. DB 마이그레이션·인프라 변경은 CTO(Theo) 확인 후 진행.
7. `CLAUDE.local.md`는 각 브랜치에 커밋되며 머지 시 덮어쓰기 방지(`.gitattributes merge=ours`).
8. **새 spec 채번 전 전 브랜치 확인 필수** — 다른 에이전트가 자기 브랜치에서 만든 spec과 번호 충돌 방지. `git fetch` 후 전 브랜치 통틀어 가장 높은 번호 +1로 채번. 충돌 발견 시 늦게 만든 쪽이 양보.
9. **핸드오프는 "내용 복붙"이 아니라 "경로 전달"** — git이 운반, 사람은 초인종:
   - 핸드오프 큐 공용 영역(예: `specs/000-backlog/`)을 두고 전원이 같은 큐를 본다.
   - 세션 시작 시 자기 핸드오프 큐를 먼저 읽는다.
   - 작업 지시·결정·완료보고는 말로 옮기지 말고 큐/문서에 적고 경로만 주고받는다.
   - **"복붙 핸드오프 카드" 표준 포맷** — 기획자가 큐 항목을 만들 때 맨 위에 아래 카드를 박는다. 대표님은 이 블록만 복사해 수신 에이전트에게 던지면 끝.
     ```
     ┌─ 핸드오프 카드 27 ──────────────────────
     [발신→수신] 실비 → 테오
     [성격]      🔴 최우선 · 신규기능 · 큰기능(spec-kit)
     [지시]      specs/027-.../spec.md 읽고, 큐 6번부터 착수
     [착수 전]   미결 #1 확정 / #2는 테오 판단
     └────────────────────────────────────────
     ```
     · **카드 고유 번호 (필수)** — 카드 상단 테두리에 **숫자만** 박는다(prefix·하이픈 없음 — 휴대폰 숫자패드로 바로 입력). 번호 = 핸드오프 큐 누적 일련번호(완료돼도 보존). 복붙 카드에도 반드시 포함 → "18 진행됐어?"로 즉시 참조. 새 카드 = 현재 큐 최대번호 +1.
     · **발신→수신**: 누가 누구에게.
     · **성격 = [긴급도][유형][범위] 3태그**: 긴급도(🔴최우선/🟡일반/🟢여유) · 유형(신규기능/버그수정/리팩토링/조사/UI개선) · 범위(큰기능 spec-kit / 가벼움 직접).
     · **지시**: 한 줄 — "어느 경로 읽고 무엇부터". 내용 복붙 금지, 경로로.
     · **착수 전**: 수신자가 막힐 결정사항(미결 번호)과 상태(확정/대기).
10. **배포 정책 — main 단일 통제 + 완료 시 통합 책임**:
    - **`main` 직접 push·배포는 CTO(Theo)만** 한다. main은 사용자가 쓰는 프로덕션 — 여러 에이전트가 동시에 건드리면 충돌·역전 사고가 난다.
    - **작업 완료분을 자기 브랜치에 커밋만 하고 방치 금지.** 환경별:
      · **로컬 워크트리(dev push 가능)**: 본인이 dev 머지까지.
      · **원격 web 세션(claude/* 격리, dev push 불가)**: 자기 브랜치 push + **핸드오프 큐 완료보고에 `[브랜치명 · 커밋해시 · "dev 흡수 필요"]` 명시** → CTO가 배포 시 흡수.
    - **공통 원칙: "완료 = 결과물이 어디 있는지 경로(브랜치·커밋)를 핸드오프에 남긴다."** 경로가 없으면 묻힌다.

## 5. 운영 철학 (제품 설계의 공통 원칙)

- **단일 진실원천(SSOT)** — 같은 정보를 두 곳에 입력하지 않는다. 한 곳에 입력하면 나머지는 파생.
- **격리 우선** — 데이터·인증·배포는 분리. 한 제품 장애가 다른 제품에 전파되지 않게.
- **중복 제거** — 같은 기능을 제품마다 새로 만들지 말고 공통 자산으로.
- **되돌릴 수 있게** — 비가역적 결정(통합·삭제)은 신중. 옵션을 죽이지 않는다.
- **호칭 ≠ 분류** — 사람·사물을 부르는 말과 시스템 분류 기준을 섞지 않는다.

---

## 6. 원격(web) 세션 / 마이그레이션 안내 규칙

- **사용자는 원격 환경이라 로컬 파일을 직접 못 연다.** DB 마이그레이션 등이 필요하면 파일 경로만 안내하지 말고 **SQL/내용 전체를 대화창에 코드블록으로 직접 출력**한다.
- 원격 web 세션은 작업 브랜치가 `claude/*`로 격리됨 → dev 직접 push 불가. §4-10 원격 흐름(경로 명시 → CTO 흡수) 따른다.

## 7. 동기화 프로토콜 (A 방식 — 양쪽 각각 반영)

> 제품 레포가 분리돼 있어 한 에이전트가 양쪽을 못 만진다. 이 PLAYBOOK이 바뀌면 **각 제품 레포에 같은 변경을 따로 반영**한다.

**바뀔 때마다:**
1. 변경자가 **원본(EO `PLAYBOOK.md`)** 을 수정 + 상단 버전(날짜·세션) 갱신.
2. **"PLAYBOOK 동기화" 카드**를 만들어 각 제품 에이전트에게 복붙:

   ```
   ┌─ 핸드오프 카드 NN ──────────────────────
   [발신→수신] 실비 → 다온
   [성격]      🟡 일반 · 동기화 · 가벼움
   [지시]      EO PLAYBOOK.md vYYYY-MM-DD 변경분을 Carelog PLAYBOOK.md에 반영. 변경 요약: (한 줄)
   └────────────────────────────────────────
   ```
3. 각 에이전트가 자기 레포 `PLAYBOOK.md`에 반영.
4. **버전(날짜)으로 최신 여부 확인** — 두 레포의 PLAYBOOK 상단 버전이 같으면 동기화 완료.

> ⚠️ **변경이 잦으면(현재 작업속도 = 하루 여러 번)** 매번 카드 던지기가 번거로울 수 있다. 그럴 땐 **변경분을 모았다가 하루 1~2회 일괄 반영**하거나, 장기적으로 git submodule/자동화로 전환 검토. 단 자동화는 운영 복잡도가 늘어 현재는 수동(A) 유지.
